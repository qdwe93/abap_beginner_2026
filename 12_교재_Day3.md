## U16. Function Module (09:15~10:15)

### 🎯 목표
- 함수 그룹과 Function Module(FM)을 만들 수 있다
- IMPORT/EXPORT/EXCEPTIONS를 정의하고, 호출 측에서 예외를 처리할 수 있다

### 📖 개념

**Function Module** = 시스템 어디서든 호출할 수 있는 **공용 함수**. 어제 쓴 `REUSE_ALV_GRID_DISPLAY`도, 오늘 쓸 팝업도 전부 누군가 만들어 둔 FM입니다. 오늘은 우리가 만드는 쪽이 됩니다.

- **함수 그룹** = FM들을 담는 상자 (FM은 반드시 그룹 안에)
- 인터페이스: **IMPORTING**(받는 값) / **EXPORTING**(주는 값) / TABLES·CHANGING(참고만) / **EXCEPTIONS**(실패를 알리는 이름들)
- 호출한 쪽에서는 예외가 번호로 떨어짐 → `sy-subrc`로 분기
- 왜 FM으로 만들까? "주문 입력값 검증"은 등록 팝업에서도, 나중에 인터페이스에서도 쓸 로직 → **한 곳에 두고 재사용**

### 💻 따라치기 1 — 함수 그룹과 FM 만들기

**(1) 함수 그룹 `ZEDU##_FG`**

1. `SE37` → 메뉴 Goto → Function Groups → **Create Group**
2. Function group `ZEDU##_FG`, Short text `## 주문관리 함수들` → [Save] → Local Object 📷

**(2) FM `ZEDU##_ORDER_CHECK` — 주문 입력값 검증**

1. SE37 초기화면: Function Module `ZEDU##_ORDER_CHECK` → [Create] → Function group `ZEDU##_FG`, Short text `주문 입력값 검증`
2. **Import 탭**:

| Parameter | Type spec. | Associated Type |
|-----------|-----------|-----------------|
| IV_CUST_ID | TYPE | ZEDU_CUSTID |
| IV_PROD_ID | TYPE | ZEDU_PRODID |
| IV_QUANTITY | TYPE | ZEDU_QTY |

3. **Export 탭**: `EV_PRICE` TYPE `ZEDU_PRICE`
4. **Exceptions 탭**: `INVALID_CUST`(고객 없음), `INVALID_PROD`(제품 없음), `INVALID_QTY`(수량 오류)
5. **Source code 탭**:

```abap
FUNCTION zedu##_order_check.

  DATA lv_cust TYPE zedu_custid.

* 1) 고객 존재 확인
  SELECT SINGLE cust_id FROM zedu_cust INTO lv_cust
    WHERE cust_id = iv_cust_id.
  IF sy-subrc <> 0.
    RAISE invalid_cust.        " 예외를 발생시키고 즉시 복귀
  ENDIF.

* 2) 제품 존재 확인 + 단가 반환
  SELECT SINGLE price FROM zedu_prod INTO ev_price
    WHERE prod_id = iv_prod_id.
  IF sy-subrc <> 0.
    RAISE invalid_prod.
  ENDIF.

* 3) 수량 검증
  IF iv_quantity <= 0.
    RAISE invalid_qty.
  ENDIF.

ENDFUNCTION.
```

6. 저장 → 활성화
7. **F8(단독 테스트)** — FM의 강력한 장점: 프로그램 없이 바로 실험 📷

| 입력 | 기대 결과 |
|------|-----------|
| C00001 / P0000003 / 3 | EV_PRICE = 45000.00 |
| C99999 / P0000003 / 3 | 예외 INVALID_CUST |
| C00001 / P0000003 / 0 | 예외 INVALID_QTY |

### 💻 따라치기 2 — 프로그램에서 호출 (`ZEDU##_U16`)

에디터에서 **[Pattern] 버튼(Ctrl+F6)** → CALL FUNCTION에 `ZEDU##_ORDER_CHECK` 입력하면 호출 골격이 자동 생성됩니다. 완성:

```abap
REPORT zedu##_u16.

PARAMETERS: p_cust TYPE zedu_custid DEFAULT 'C00001',
            p_prod TYPE zedu_prodid DEFAULT 'P0000003',
            p_qty  TYPE zedu_qty    DEFAULT 3.

DATA: gv_price  TYPE zedu_price,
      gv_amount TYPE zedu_amount.

START-OF-SELECTION.

  CALL FUNCTION 'ZEDU##_ORDER_CHECK'
    EXPORTING
      iv_cust_id   = p_cust        " 내가 '주는' 값 (FM 입장에선 IMPORT)
      iv_prod_id   = p_prod
      iv_quantity  = p_qty
    IMPORTING
      ev_price     = gv_price      " 내가 '받는' 값
    EXCEPTIONS
      invalid_cust = 1             " 예외 -> 번호 매핑
      invalid_prod = 2
      invalid_qty  = 3
      OTHERS       = 4.

  CASE sy-subrc.
    WHEN 0.
      gv_amount = gv_price * p_qty.
      WRITE: / '검증 통과! 단가:', gv_price, '/ 예상 금액:', gv_amount.
    WHEN 1.
      WRITE: / '오류: 고객이 존재하지 않습니다 ->', p_cust.
    WHEN 2.
      WRITE: / '오류: 제품이 존재하지 않습니다 ->', p_prod.
    WHEN 3.
      WRITE / '오류: 수량은 1 이상이어야 합니다.'.
    WHEN OTHERS.
      WRITE / '알 수 없는 오류'.
  ENDCASE.
```

> 💡 호출측 EXPORTING/IMPORTING이 FM 정의와 **반대 방향**인 게 처음엔 헷갈립니다. "내가 주는 것 EXPORTING, 내가 받는 것 IMPORTING" — 항상 호출자 시점입니다.

> 📦 **디버거 F5 vs F6의 차이를 볼 시간**: CALL FUNCTION 줄에 브레이크포인트 → F5는 FM **안으로** 들어가고, F6은 **건너뜁니다**. 직접 확인해보세요.

## U17. DB CRUD — INSERT/UPDATE/MODIFY/DELETE (10:30~11:30)

### 🎯 목표
- Open SQL로 DB에 쓰고(INSERT), 고치고(UPDATE/MODIFY), 지울(DELETE) 수 있다
- COMMIT WORK / ROLLBACK WORK의 의미를 안다
- **변경 후 SE16N으로 확인하는 습관**을 들인다

### 📖 개념

| 문장 | 동작 | 실패 시(sy-subrc≠0) |
|------|------|---------------------|
| `INSERT 테이블 FROM 구조.` | 새 행 추가 | 키 중복이면 4 |
| `UPDATE 테이블 FROM 구조.` | 키로 찾아 전체 갱신 | 키 없으면 4 |
| `UPDATE 테이블 SET 필드=값 WHERE ...` | 조건 행들의 특정 필드만 갱신 | 대상 0건이면 4 |
| `MODIFY 테이블 FROM 구조.` | 있으면 UPDATE, 없으면 INSERT | |
| `DELETE FROM 테이블 WHERE ...` | 조건 행 삭제 | 대상 0건이면 4 |

- `sy-dbcnt` = 방금 처리된 행 수
- **COMMIT WORK** = 지금까지의 변경을 DB에 확정 / **ROLLBACK WORK** = 미확정 변경 취소.
  (화면이 바뀔 때 등 암묵적 커밋도 일어나지만, **"변경했으면 명시적으로 COMMIT"**을 3일 과정의 규칙으로 삼습니다)
- ⚠️ **절대 금지**: SAP 표준 테이블에 직접 INSERT/UPDATE/DELETE. 정합성이 깨지고 장애로 이어집니다. 표준 데이터는 표준 함수(BAPI)로만. 우리는 **우리가 만든 Z테이블**만 만집니다.

### 💻 따라치기 — `ZEDU##_U17` (3라운드로 나눠 실행)

**라운드 1**: 여기까지만 입력하고 실행 → **SE16N으로 9000000001 확인** (연습용 번호대라 샘플 20건과 안 섞임)

```abap
REPORT zedu##_u17.

DATA: gs_order TYPE zedu##_order.

START-OF-SELECTION.

* 1) INSERT
  CLEAR gs_order.
  gs_order-order_id   = 9000000001.
  gs_order-cust_id    = 'C00002'.
  gs_order-prod_id    = 'P0000005'.
  gs_order-quantity   = 2.
  gs_order-status     = 'N'.
  gs_order-order_date = sy-datum.
  gs_order-remark     = 'CRUD 실습'.
  gs_order-ernam      = sy-uname.

  INSERT zedu##_order FROM gs_order.
  WRITE: / '1) INSERT sy-subrc =', sy-subrc.        " 0
  COMMIT WORK.
```

**라운드 2**: 아래를 이어붙여 다시 실행 → 1)의 subrc가 이번엔 4 (이미 있으니까!) → SE16N에서 수량 99, 상태 C 확인

```abap
* 2) UPDATE 방법 A: 구조로 (키로 찾아 전체 교체)
  SELECT SINGLE * FROM zedu##_order INTO gs_order
    WHERE order_id = 9000000001.
  IF sy-subrc <> 0.
    MESSAGE '수정할 주문이 없습니다. 라운드 1부터 실행하세요.' TYPE 'S' DISPLAY LIKE 'E'.
    RETURN.
  ENDIF.
  gs_order-quantity = 99.
  UPDATE zedu##_order FROM gs_order.
  WRITE: / '2) UPDATE sy-subrc =', sy-subrc.
  COMMIT WORK.

* 3) UPDATE 방법 B: SET ... WHERE (필드만 콕 집어)
  UPDATE zedu##_order SET status = 'C'
    WHERE order_id = 9000000001.
  WRITE: / '3) UPDATE SET 처리 건수 =', sy-dbcnt.
  COMMIT WORK.
```

**라운드 3**: 마무리 — MODIFY / ROLLBACK / DELETE

```abap
* 4) MODIFY: 없으면 INSERT, 있으면 UPDATE
  CLEAR gs_order.
  gs_order-order_id   = 9000000002.
  gs_order-cust_id    = 'C00001'.
  gs_order-prod_id    = 'P0000001'.
  gs_order-quantity   = 1.
  gs_order-status     = 'N'.
  gs_order-order_date = sy-datum.
  gs_order-ernam      = sy-uname.
  MODIFY zedu##_order FROM gs_order.
  WRITE: / '4) MODIFY sy-subrc =', sy-subrc.
  COMMIT WORK.

* 5) ROLLBACK 체험: 갱신했다가 무르기
  UPDATE zedu##_order SET quantity = 777
    WHERE order_id = 9000000002.
  WRITE: / '5) UPDATE(커밋 전) sy-subrc =', sy-subrc.
  ROLLBACK WORK.                    " 확정 전이므로 취소됨
  " -> SE16N: 9000000002의 수량은 여전히 1 !

* 6) DELETE
  DELETE FROM zedu##_order WHERE order_id = 9000000001.
  WRITE: / '6) DELETE 처리 건수 =', sy-dbcnt.
  COMMIT WORK.
```

> 💡 **SE16N을 두 번째 세션에 상시 대기**시키세요 (`/o SE16N`). 변경 → 확인 → 변경 → 확인. 오늘 오후 내내 이 리듬입니다.

## U18. MESSAGE와 메시지 클래스 (11:30~12:00)

### 🎯 목표
- 메시지 타입(S/I/W/E/A)의 차이를 안다
- SE91 메시지 클래스를 만들어 MESSAGE 문으로 사용할 수 있다

### 📖 개념

| 타입 | 표시 | 흐름 |
|------|------|------|
| **S** | 하단 상태바 (초록) | 계속 진행 |
| **I** | 팝업 | 확인 후 계속 |
| **W** | 상태바 (노랑) | 위치 따라 다름 |
| **E** | 상태바 (빨강) | **그 자리에서 처리 중단** (선택화면에선 화면 유지 — U14에서 체험) |
| **A** | 팝업 | 프로그램 강제 종료 |

두 가지 사용법:
```abap
MESSAGE '수량을 확인하세요' TYPE 'E'.          " ① 리터럴 (임시·테스트용)
MESSAGE e012(zedu##_msg).                      " ② 메시지 클래스 (실무 표준)
MESSAGE s001(zedu##_msg) WITH lv_order_id.     "    &1 자리에 값 채우기
```
왜 클래스로? 문구 재사용·일괄 관리·다국어 번역 가능. **"같은 문구를 두 번 쓰게 되면 메시지 클래스로"**.

**오늘의 핵심 기술** — `DISPLAY LIKE`:
```abap
MESSAGE s020(zedu##_msg) DISPLAY LIKE 'E'.     " 빨갛게 보이지만(E처럼) 흐름은 계속(S)
```
잠시 후 ALV 버튼 처리(콜백) 안에서 E를 그대로 쓰면 흐름이 이상해질 수 있어, **"S로 보내되 E처럼 보이게"**가 현장의 관용구입니다. 오후에 계속 씁니다.

### 💻 따라치기 — 메시지 클래스 `ZEDU##_MSG` 만들기

1. `SE91` → Message class `ZEDU##_MSG` → [Create] 📷
2. 메시지 입력 (오후에 전부 사용합니다):

| 번호 | 메시지 텍스트 |
|------|---------------|
| 001 | 주문 &1이(가) 등록되었습니다. |
| 002 | 주문 &1이(가) 수정되었습니다. |
| 003 | 주문 &1이(가) 삭제되었습니다. |
| 010 | 존재하지 않는 고객입니다: &1 |
| 011 | 존재하지 않는 제품입니다: &1 |
| 012 | 수량은 1 이상이어야 합니다. |
| 013 | 입력할 수 없는 상태값입니다: &1 |
| 020 | 완료된 주문은 수정할 수 없습니다. |
| 021 | 신규 상태의 주문만 삭제할 수 있습니다. |
| 022 | 처리할 행에 커서를 두고 버튼을 누르세요. |

3. 저장. 테스트 프로그램 `ZEDU##_U18`:

```abap
REPORT zedu##_u18.

PARAMETERS p_qty TYPE zedu_qty DEFAULT 0.

START-OF-SELECTION.

  IF p_qty <= 0.
    MESSAGE s012(zedu##_msg) DISPLAY LIKE 'E'.   " 빨갛지만 중단 없음
    WRITE / '(E였다면 이 줄은 실행되지 않았을 것)'.
  ENDIF.

  MESSAGE s001(zedu##_msg) WITH '0000000099'.    " &1 채우기
  WRITE / '처리 끝'.
```

수량 0으로 실행 → 빨간 메시지 + 아랫줄 실행됨을 확인. `DISPLAY LIKE 'E'`를 지우고 `MESSAGE e012(...)`로 바꿔 실행 → 아랫줄이 실행되지 않음. **이 차이가 오후 버튼 로직의 생명줄**입니다.

## U19. PF-STATUS와 버튼 이벤트 (13:00~13:50)

### 🎯 목표
- ALV 툴바에 커스텀 버튼을 추가할 수 있다 (PF-STATUS)
- 버튼 클릭을 USER_COMMAND 콜백에서 받아 처리할 수 있다
- 커서가 놓인 ALV 행을 읽을 수 있다

### 📖 개념

버튼을 다는 데 필요한 두 조각:

1. **PF-STATUS(GUI Status)** = 이 화면에 어떤 버튼·메뉴가 있는지 정의. 각 버튼에는 **기능 코드(Function Code)** 라는 이름표
2. **USER_COMMAND 콜백** = 버튼이 눌리면 ALV가 호출해주는 FORM. 파라미터로 **어떤 버튼이었는지**(기능 코드)와 **커서가 어디 있었는지**(`slis_selfield-tabindex` = 내부 테이블 행 번호)가 들어옴

> 📦 **용어 구분**: 고전 리스트의 `AT USER-COMMAND` 이벤트와 ALV의 `USER_COMMAND` **콜백**은 다른 진입점입니다. 이 과정은 ALV 콜백 방식을 실습합니다.

```
[사용자가 '수정' 클릭] → ALV가 FORM user_command 호출
                          pv_ucomm    = 'FC_EDIT'   (뭘 눌렀나)
                          ps_selfield-tabindex = 7  (커서는 7행)
```

### 💻 따라치기 — `ZEDU##_U19`

**(1) 어제 프로그램 복사**: SE38 → `ZEDU##_LIST` → 복사(Ctrl+F5) → `ZEDU##_U19` (**Text elements 체크**) → 활성화·실행 확인

**(2) ALV 호출에 콜백 2줄 추가** (FORM show_alv 안):

```abap
  CALL FUNCTION 'REUSE_ALV_GRID_DISPLAY'
    EXPORTING
      i_callback_program       = sy-repid
      i_callback_pf_status_set = 'SET_STATUS'     " ★ 추가: 상태 세팅 FORM 이름
      i_callback_user_command  = 'USER_COMMAND'   " ★ 추가: 버튼 처리 FORM 이름
      i_structure_name         = 'ZEDU##_S_LIST'
      ...이하 동일...
```

**(3) 프로그램 맨 아래에 FORM 2개 추가**:

*&---------------------------------------------------------------------*
*& Report ZEDU##_MAIN — 간이 주문 관리 (Toy Project 완성본)
*&---------------------------------------------------------------------*
REPORT zedu##_main.

TYPE-POOLS: slis.

*----- 전역 데이터 ------------------------------------------------------
DATA: gt_list  TYPE TABLE OF zedu##_s_list,
      gs_list  TYPE zedu##_s_list,
      gs_order TYPE zedu##_order.

*----- 선택화면 ---------------------------------------------------------
SELECT-OPTIONS: s_ordid FOR gs_order-order_id,
                s_cust  FOR gs_order-cust_id,
                s_date  FOR gs_order-order_date.
PARAMETERS: p_stat TYPE zedu##_status.

*----- 이벤트 -----------------------------------------------------------
INITIALIZATION.
  CLEAR s_date.
  s_date-sign   = 'I'.
  s_date-option = 'BT'.
  s_date-low    = sy-datum - 90.
  s_date-high   = sy-datum.
  APPEND s_date.

START-OF-SELECTION.
  PERFORM get_data.
  PERFORM show_alv.

*&---------------------------------------------------------------------*
*& 조회

FORM get_data.
  SELECT a~order_id a~cust_id b~cust_name
         a~prod_id  c~prod_name c~price
         a~quantity a~status a~order_date a~remark a~ernam
    FROM zedu##_order AS a
    INNER JOIN zedu_cust AS b ON a~cust_id = b~cust_id
    INNER JOIN zedu_prod AS c ON a~prod_id = c~prod_id
    INTO CORRESPONDING FIELDS OF TABLE gt_list
    WHERE a~order_id   IN s_ordid
      AND a~cust_id    IN s_cust
      AND a~order_date IN s_date.

  IF p_stat IS NOT INITIAL.
    DELETE gt_list WHERE status <> p_stat.
  ENDIF.

  LOOP AT gt_list INTO gs_list.
    gs_list-amount = gs_list-quantity * gs_list-price.
    MODIFY gt_list FROM gs_list TRANSPORTING amount.
  ENDLOOP.

  SORT gt_list BY order_id.
ENDFORM.

FORM change_order USING pv_index TYPE sytabix.

  ls_field-value = gs_list-status.
  APPEND ls_field TO lt_fields.

  CALL FUNCTION 'POPUP_GET_VALUES'
    EXPORTING
      popup_title = '주문 수정'

      returncode  = lv_rc

      fields      = lt_fields

  IF sy-subrc <> 0 OR lv_rc = 'A'.
    RETURN.
  ENDIF.

  READ TABLE lt_fields INTO ls_field INDEX 1.  lv_qty    = ls_field-value.
  READ TABLE lt_fields INTO ls_field INDEX 2.  lv_status = ls_field-value.

  " TODO C: 값 검증
  IF lv_qty <= 0.
    MESSAGE s012(zedu##_msg) DISPLAY LIKE 'E'.

  UPDATE zedu##_order

ENDFORM.
