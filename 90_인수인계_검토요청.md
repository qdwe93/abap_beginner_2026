# 인수인계 & 전체 검토 요청서

> 이 문서는 **새로운 세션(검토자)** 에게 프로젝트 전체를 인수인계하고 검토를 요청하는 문서입니다.
> 검토자는 이 문서를 끝까지 읽은 뒤, 맨 아래 "검토 수행 지침"에 따라 작업하세요.

## 1. 프로젝트 개요

- **목적**: SAP ABAP 기초교육 **3일 오프라인 실습 과정**의 커리큘럼·교재·퀴즈·강사자료 개발
- **환경**: S/4HANA 2020(온프레미스, "2003"으로 언급됐으나 2020 추정 — 미확정), SAP GUI, **Classic ABAP** (Cloud/BTP/RAP 제외)
- **문법 정책**: 구문법이 주 커리큘럼. 7.40+ 신문법·new Open SQL은 각 유닛 말미 "✨ 모던 ABAP" 박스(보조 커리큘럼)
- **교수법 원칙**: 유닛마다 `개념(20~30분) → 따라치기 → 퀴즈` 사이클. 스켈레톤 제공 + 학습 포인트만 직접 타이핑
- **최종 목표**: Day 3 오후에 toy project **"간이 주문 관리 프로그램"** (조회+ALV+등록/수정/삭제 버튼) 완성
- **수강생**: 프로그래밍 경험자·초보 혼재 / 매일 09:00~18:00 (점심 1h, 실질 8h)

## 3. 핵심 설계 결정 (검토 시 전제로 삼을 것 — 뒤집지 말 것)

1. **공용 마스터 + 학생별 주문 테이블**: 고객(ZEDU_CUST)·제품(ZEDU_PROD)은 강사 사전 생성 공용, 주문(ZEDU##_ORDER)만 학생이 Day 1에 직접 생성. `##` = 학생 번호 2자리
2. **주문 상태 3종 N/C/X**: 취소(X)는 도메인·샘플 데이터에만 존재. **취소 처리 기능은 확장 과제** (기본 과정 아님) — 사용자 확정 사항
3. **샘플 데이터**: 학생당 주문 20건, `ZEDU_DATA_LOAD`(강사 제공)로 투입, **초기화(리셋) 옵션 포함** — 사용자 확정 사항
4. ALV는 `REUSE_ALV_GRID_DISPLAY` + `i_structure_name`(DDIC 스트럭처 ZEDU##_S_LIST, 전 필드 DE 참조 — 컬럼 제목 자동화 목적)
5. 입력 팝업은 표준 FM `POPUP_GET_VALUES` (Dynpro 미학습 상태 고려)
6. 프로그램 복사 체인: `ZEDU##_LIST`(Day2 종합실습) → 복사 → `ZEDU##_U19`(버튼 추가) → 복사 → `ZEDU##_MAIN`(TP에서 로직 구현)
7. 단순화 결정 6건(채번 MAX+1, DEC 단가 등)은 02 문서 7절에 "실무에서는" 대비표로 정리되어 있고 강사 가이드(STEP 5)에서 보충 예정 — 교재에서 고칠 대상 아님

## 4. 이미 알려진 리스크 (검토에서 재확인 요망)

- ABAP 코드는 **실제 시스템에서 미검증**. 특히: PF-STATUS 복사 경로(SE41, SAPLKKBL의 STANDARD_FULLSCREEN), POPUP_GET_VALUES의 라벨/타입 처리, REUSE_ALV 파라미터 철자
- S/4HANA 정확 버전 미확정 (신문법 범위에 경미한 영향)
- 스크린샷(📷)은 위치 표시만 있고 실물 없음 (의도된 상태 — 강사가 시스템에서 캡처 예정)

## 5. 남은 작업 (검토 대상 아님, 참고용)

- STEP 4: 추가 퀴즈집 (유닛별 쉬움/심화 변형 + 해설)
- STEP 5: 강사용 가이드 (진행 팁, T-code 모음, 운영 상식, FAE 함정 등 심화 설명)
- STEP 6: 패키징 (.abap 파일 분리, 강사 사전 준비 체크리스트, ZEDU_DATA_LOAD 등 강사용 프로그램 소스)

## 6. 검토 수행 지침 (검토자용)

### 검토 항목 (우선순위 순)

**A. ABAP 코드 정확성** 🔴 최우선
- 교재의 모든 코드 블록을 classic ABAP(S/4 2020) 문법으로 정밀 검토: 문장 구조, SELECT 절 순서, 파라미터 철자(REUSE_ALV_GRID_DISPLAY, POPUP_GET_VALUES, POPUP_TO_CONFIRM), 타입 정합성(sytabix, sval, slis_*), sy-subrc/sy-tabix 사용 시점
- 특히 Day3 부록 B의 `ZEDU##_MAIN` 완성 코드 전문 — 이 코드가 틀리면 강의 마지막이 무너짐
- 퀴즈 정답 코드가 실제로 기대 결과를 내는지 (예: Day1 Q05의 1683, U04의 55)

**B. 문서 간 일관성** 🔴
- 02의 스펙 ↔ 교재 사용처 대조: 테이블/스트럭처 필드 구성(U07 표), 공용 DE 9종 이름, FM 인터페이스(02 3-3 ↔ Day3 U16 ↔ MAIN 호출부), 메시지 번호·텍스트(02 3-4 ↔ Day3 U18 표 ↔ 코드의 MESSAGE 호출 전부)
- 01 시간표의 유닛 시간 ↔ 교재 각 유닛 헤더의 시간 표기
- 프로그램 이름 목록(01 유닛 표 ↔ 교재 본문), 퀴즈 번호 체계(Day1 Q01~Q08, Day2 R2·Q09~Q15, Day3 R3·Q16~Q19)
- 샘플 데이터(Day1 부록 B) ↔ 교재 예제에서 참조하는 값들(P0000001 단가 등)

**C. 요구사항 커버리지** 🟡
- 00 문서 4절의 학습 목표 9개가 유닛 어딘가에서 실제로 다뤄지는가
- 00 문서 7절 보강 사항(디버거/ALV/MESSAGE/복습퀴즈+SE16N)이 반영됐는가

**D. 교수법·난이도** 🟡
- 각 유닛이 "개념→따라치기→퀴즈" 구조를 지키는가, 개념 설명이 초보 기준으로 비약 없는가
- 유닛 분량이 배정 시간(01 시간표) 안에 현실적으로 들어가는가 — 특히 U07(테이블+스트럭처 50분), U19(PF-STATUS 50분), TP-2(50분)
- 스켈레톤 원칙: 퀴즈가 "배운 것만으로" 풀리는가, TODO 범위가 적절한가
- ✨ 박스가 주 커리큘럼을 침범하지 않는가

**E. 운영 관점** 🟢
- 강사 사전 준비물이 교재에서 암묵적으로 더 늘어나지 않았는가 (02 6절과 대조)
- 학생이 막힐 만한 지점에 안전장치(팁 박스, 리셋 안내)가 있는가

### 산출물 형식

`90_검토보고서.md` 파일 하나로 작성:

1. **총평** (5줄 이내)
2. **이슈 목록** — 심각도별 정렬, 각 이슈는: `[심각도] 파일명 § 위치 | 문제 | 근거 | 수정 제안`
   - 🔴 블로커: 코드 오동작·강의 진행 불가·스펙 모순
   - 🟡 중요: 학생 혼란 유발·시간 초과 위험
   - 🟢 개선: 선택적 품질 향상
3. **일관성 대조표** — B 항목의 대조 결과 (일치/불일치 표)
4. **시간 현실성 평가** — 위험 유닛 top 3와 근거
5. **총괄 판정**: 이대로 강의 가능 여부 + 최소 수정 목록

### 금지 사항

- 교재·설계 문서를 **직접 수정하지 말 것** (보고서만 작성)
- 3절의 확정 설계를 뒤집는 제안 금지 (개선 의견은 🟢로만)
- 스크린샷 부재, STEP 4~6 미완성을 이슈로 올리지 말 것 (알려진 상태)
