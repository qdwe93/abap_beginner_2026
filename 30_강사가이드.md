## 1. 과정 운영 원칙

### 1-1. 이 과정의 결승선

학생이 3일 안에 다음 흐름을 직접 연결하는 것이 목표다.

```text
DDIC 설계 → 내부 테이블 → Open SQL/JOIN → 선택화면 → ALV
→ 검증 FM → DB CRUD → 메시지 → PF-STATUS/ALV 콜백
→ 주문 조회·등록·수정·삭제 통합
```

모든 문법을 넓게 소개하는 것보다 이 한 바퀴를 완주하는 것이 우선이다. 학생이 세부 문법을 잊어도 “어디에서 정의하고, 어디에서 조회하고, 어디에서 검증하며, 어디에서 DB를 바꾸는가”를 설명할 수 있으면 성공이다.

### 1-2. 강의 중 지킬 규칙

1. 강사가 먼저 완성 코드를 길게 타이핑하지 않는다. 스켈레톤을 열고 TODO 한 덩어리씩 시연한다.
2. 한 TODO가 끝날 때마다 저장 → 활성화 → 실행한다. 마지막에 한꺼번에 디버깅하지 않는다.
3. DB 변경 뒤에는 SE16N 확인을 한 세트로 묶는다.
4. 학생의 `##` 번호와 객체명을 먼저 확인한 뒤 코드 오류를 본다.
5. 3분 이상 같은 화면에서 전원이 막히면 개인 오류가 아니라 환경·설명 문제로 간주하고 전체 화면을 멈춘다.
6. 정해진 중간 마감시점이 오면 미완료 학생을 기준본으로 전환한다. 완주 경험을 보호하는 조치다.
7. 본문은 classic 문법으로 진행하고 ✨ 박스는 시간이 남을 때만 비교한다.
8. 취소(X)는 기본 과정에서 조회·필터용이다. 기본 구현은 X 수정 차단까지만 하며 취소 버튼은 TP-5 확장 과제로 둔다.

### 1-3. 혼합 수준 반 운영

| 학생 상태 | 강사 대응 |
|-----------|-----------|
| 따라오지 못함 | 해당 유닛의 `20_퀴즈집.md` 쉬운 변형(E)을 본문 대신 또는 재연습으로 제공 |
| 정상 진도 | 본 교재 퀴즈까지 수행 |
| 빨리 끝남 | 심화 변형(D)을 제공하되 다음 본문을 미리 타이핑하게 하지 않음 |
| 오타가 반복됨 | 완성 코드를 주기 전에 활성화 오류의 첫 줄, 객체명, 마침표, 따옴표를 함께 확인 |
| 환경/권한 문제 | 즉시 기준본·강사 지원 경로로 전환하고 수업 후 원인 분석 |

빠른 학생에게 옆 학생의 코드를 직접 고치게 하지 않는다. 설명은 허용하되 키보드는 원래 학생이 잡게 한다.

## OT. 시스템 접속과 SAP/ERP 소개 (09:00~09:40)

### 진행 포인트

- 10분: 과정 목표와 최종 프로그램 시연 이미지
- 10분: SAP GUI, 트랜잭션, 클라이언트, 사용자 개념
- 10분: 학생 번호·`##`·Z 네임스페이스·`$TMP`
- 10분: 접속 및 `/nSE38`, `/oSE16N` 확인

“SAP는 화면 묶음”이 아니라 업무 데이터와 프로세스를 통합하는 시스템이며 ABAP은 그 시스템 안에서 실행되는 언어라고 설명한다.

### 자주 막힘

- 다른 클라이언트 접속: 같은 계정처럼 보여도 데이터가 다르다.
- 사용자 메뉴에서 T-code를 못 찾음: 명령창에 직접 입력하는 법을 먼저 가르친다.
- `/n`과 `/o` 혼동: `/n`은 현재 세션 이동, `/o`는 새 세션이다.

### 예상 질문

**Q. SAP GUI와 SAP 서버는 같은 것인가요?**
A. GUI는 입력·표시를 담당하는 클라이언트이고 프로그램과 데이터는 서버에서 처리된다.

**Q. 왜 이름 앞에 Z를 붙이나요?**
A. SAP 표준 객체와 고객 개발 객체를 구분하는 고객 네임스페이스다. 이 과정의 `ZEDU`는 교육용 접두어다.

## U01. SE38 첫 프로그램 (09:40~10:30)

### 시간 배분

- 10분: 프로그램 객체, 편집기, 저장/활성/실행 구분
- 20분: 첫 프로그램 따라치기
- 10분: 오류 체험(마침표·따옴표)
- 10분: Q01 또는 E01

### 강조

- 저장은 소스 저장, 활성화는 실행 가능한 버전 생성이다.
- ABAP 문장은 마침표로 끝난다.
- 대소문자는 키워드 의미에 영향이 적지만 문자열은 구분된다.

### 자주 막힘과 대처

| 증상 | 확인 순서 |
|------|-----------|
| 실행해도 옛 결과 | 활성화했는지 확인 |
| 빨간 문법 오류 | 오류 목록 첫 줄 더블클릭 → 앞 문장의 마침표 확인 |
| 객체 생성 안 됨 | `ZEDU##_...`에서 ##를 실제 번호로 바꿨는지 확인 |

### 예상 질문

**Q. 저장만 하면 왜 실행이 안 바뀌나요?**
A. 실행은 마지막 활성 버전을 사용한다. 저장 중인 소스와 활성 런타임 버전이 분리되어 있다.

### 강조

- `C`와 `N`은 숫자처럼 보여도 문자 성격이고, 계산은 `I`/`P`를 쓴다.

## U03. 조건문 (13:00~13:50)

### 진행 팁

- 점심 직후 3분 동안 상태 N/C/X를 손으로 말하게 한다.
- IF는 조건식, CASE는 한 값의 여러 갈래라는 기준을 반복한다.
- 조건 순서가 결과를 바꾸는 예를 보여준다. VIP 조건을 일반 조건보다 먼저 둔다.

### 자주 막힘

- `=`와 대입을 다른 언어의 `==`와 혼동
- `ENDIF.` 또는 `ENDCASE.` 누락
- 소문자 `'n'`을 넣고 왜 N 분기로 안 가는지 질문

### 예상 질문

**Q. CASE에 범위 조건도 쓸 수 있나요?**
A. 이 과정에서는 한 값의 고정 분기에 사용한다. 범위·복합 조건은 IF가 명확하다.

## U05. 디버거 기초 (15:05~15:45)

### 시간 배분

- 5분: 디버깅 목적과 중단점
- 15분: `/h`, F5/F6/F8, 변수창
- 10분: 변수값 변경 체험
- 10분: Q06 버그 찾기

### 강조

- F5는 내부로, F6는 한 문장 실행, F8은 다음 중단점까지다.
- “코드를 바라보기”보다 실행 중 값이 언제 달라지는지 찾는 도구다.
- 운영에서 값 변경 디버깅은 권한·감사 이슈가 있으므로 교육 시스템에서만 한다.

### 자주 막힘

- 중단점이 안 걸림: 활성 버전과 실행 프로그램 이름 확인
- 너무 깊이 들어감: F6/F7로 복귀하고 강사 화면을 따라온다.
- 시스템 변수와 일반 변수를 구분 못함: `sy-` 접두어를 표시한다.

### 예상 질문

**Q. 운영 시스템에서도 디버깅하나요?**
A. 긴급 분석에 제한적으로 사용하지만 권한·절차·감사가 필요하다. 값 변경은 특히 엄격하다.

## U12. Open SQL ② JOIN과 집계 (13:20~14:10)

### 진행 팁

- JOIN 전에 “주문에는 이름이 없다”는 문제를 먼저 제시한다.
- 별칭 `a~`, `b~`, `c~`를 표의 색과 맞춘다.
- JOIN 결과의 행 수가 줄면 마스터 누락 가능성을 설명한다.
- FOR ALL ENTRIES는 빈 원본 테이블의 위험만 소개하고 실습하지 않는다.

### 자주 막힘

- 같은 필드명을 별칭 없이 작성
- `INTO CORRESPONDING FIELDS`인데 결과 구조 필드명이 다름
- 집계하지 않은 필드를 GROUP BY에 빠뜨림
- `UP TO`와 `ORDER BY` 위치 오타

### 예상 질문

**Q. INNER JOIN에서 마스터가 없으면 주문은 나오나요?**

A. 나오지 않는다. 한쪽이 없어도 주문을 보여야 하면 LEFT OUTER JOIN을 검토한다.

**Q. FOR ALL ENTRIES가 왜 위험한가요?**

A. 원본 내부 테이블이 비었을 때 WHERE 조건이 기대와 다르게 전체 조회가 될 수 있다. 실행 전 `IS NOT INITIAL` 확인이 필수다.

## U16. Function Module (09:15~10:15)

### 시간 안전장치

- 남은 시간: 호출 프로그램과 디버거 Step Into

### 진행 순서

1. 함수 그룹은 FM의 컨테이너라고 설명한다.
2. IMPORTING/EXPORTING/EXCEPTIONS 탭을 먼저 완성한다.
3. 소스를 입력하고 활성화한다.
4. SE37 단독 테스트로 정상과 오류를 확인한다.

TABLES와 CHANGING은 인터페이스 형태만 소개하고 실습을 늘리지 않는다. 현대 인터페이스에서는 명시적 타입의 IMPORTING/EXPORTING/CHANGING을 선호하는 맥락만 말한다.

- 함수 그룹을 활성화하지 않음
- 예외 이름과 호출부 철자가 다름
- FM 소스의 자동 생성 주석/인터페이스 영역을 잘못 편집
- SE37 테스트에서 수량을 문자처럼 입력
- `OTHERS`를 수량 오류로 오진

### 예상 질문

**Q. FORM과 Function Module은 무엇이 다른가요?**

A. FORM은 주로 같은 프로그램 내부 서브루틴이고, FM은 함수 그룹에 속하며 다른 프로그램에서도 정해진 인터페이스로 호출할 수 있다.

**Q. 클래스 메서드 대신 FM을 왜 배우나요?**

A. SAP 표준 API와 기존 시스템에서 FM을 광범위하게 만나며, 인터페이스·예외·호출 개념의 좋은 입문이다. 이후 OO ABAP 메서드와 비교한다.

**Q. MODIFY는 편한데 항상 쓰면 안 되나요?**

**Q. COMMIT을 매 문장 뒤에 하면 안전한가요?**

A. 아니다. 함께 성공·실패해야 하는 업무 단위를 끊을 수 있다. 본 과정은 단순화를 위해 메인에서 명시하지만 실무는 LUW를 설계한다.

# 6. 개발·운영 실무 상식

## 6-1. DEV → QAS → PRD 랜드스케이프

```text
DEV(개발·단위 테스트)
  ↓ Transport
QAS(통합·사용자 테스트)
  ↓ 승인된 Transport
PRD(운영)
```

- 개발자는 일반적으로 PRD에서 소스를 직접 수정하지 않는다.
- 같은 객체의 이동 순서가 뒤바뀌면 이전 버전이 덮어쓸 수 있다.
- 코드만 옮기는 Workbench 요청과 클라이언트 의존 설정을 옮기는 Customizing 요청을 구분한다.
- Transport에는 객체뿐 아니라 의존 객체와 이동 순서를 함께 고려한다.

### 예상 질문

**Q. 운영에서 급하면 바로 수정하면 안 되나요?**

A. 긴급 변경도 조직의 Emergency Change 절차, 승인, 기록, 역이동 방지, 사후 동기화가 필요하다.

## 6-2. 패키지와 `$TMP`

| 항목 | `$TMP` | 정식 Z 패키지 |
|------|--------|---------------|
| 목적 | 로컬 테스트·교육 | 팀 개발·이동 |
| Transport | 일반적으로 없음 | 요청에 기록 |
| 소유/구조 | 개인·임시 성격 | 애플리케이션 계층과 책임에 따라 구성 |
| 본 과정 | 사용 | 개념 소개 |

교육 객체를 `$TMP`에 저장하는 것은 속도와 단순함을 위한 선택이다. 실무 객체는 담당 패키지, 소프트웨어 컴포넌트, Transport Layer를 확인한다.

## 6-3. 개발자 키와 권한

- 개발자 키는 개발 객체 변경 자격과 관련된 전통적 온프레미스 개념이다.
- T-code가 열리는 것과 특정 객체를 변경할 수 있는 것은 별개다.
- 권한 오류 직후 SU53을 실행해 실패한 권한 객체를 본다.
- 필요한 최소 권한만 부여하는 최소권한 원칙을 따른다.
- 교육 계정으로 표준 객체 수정을 허용하지 않는다.

## 6-4. 네이밍 컨벤션

프로젝트 표준이 있으면 그것이 최우선이다. 일반적으로 다음을 구분하면 읽기 쉽다.

| 접두어 예 | 의미 |
|-----------|------|
| `gv_`, `gs_`, `gt_` | 전역 값·구조·내부 테이블 |
| `lv_`, `ls_`, `lt_` | 로컬 값·구조·내부 테이블 |
| `iv_`, `ev_`, `cv_` | Importing·Exporting·Changing 파라미터 |
| `gc_`, `lc_` | 전역·로컬 상수 |
| `ty_` | 로컬 타입 |

이 과정은 교육용으로 익숙한 헝가리식 접두어를 사용한다. 최신 팀에서는 Clean ABAP 등 다른 규칙을 쓸 수 있으므로 조직 표준을 따른다.

## 6-5. 트랜잭션과 LUW

- 한 업무 단위에서 함께 성공해야 하는 변경을 묶는다.
- 중간 COMMIT은 앞부분만 확정되어 되돌리기 어려운 부분 성공을 만들 수 있다.
- BAPI는 반환 메시지를 확인한 뒤 `BAPI_TRANSACTION_COMMIT` 또는 ROLLBACK 패턴을 따르는 경우가 많다.
- Update Task, RFC, Background Job은 각자의 트랜잭션 경계를 이해해야 한다.

## 6-6. 변경 추적과 감사

- 누가 언제 무엇을 바꿨는지 추적할 필드와 변경문서가 필요한지 설계한다.
- `sy-uname`, 날짜만 저장한다고 완전한 감사 추적이 되는 것은 아니다.
- 개인정보·재무 데이터는 조회 권한, 로그, 마스킹, 보존정책까지 고려한다.

# 7. 반드시 조심할 코드 패턴

## 7-1. `SELECT *` 남용

### 위험

- 필요 없는 열 전송
- 구조 변경의 영향 확대
- 코드 의도가 불명확

### 지도 문장

교육 초반에는 구조를 단순화하려고 `SELECT *`를 쓰지만, 목록 프로그램에서는 필요한 필드만 명시하는 습관을 들인다.

## 7-2. LOOP 안의 SELECT

```abap
LOOP AT gt_order INTO gs_order.
  SELECT SINGLE ...                    " 반복 횟수만큼 DB 접근
ENDLOOP.
```

N+1 접근 패턴이 되어 데이터가 커질수록 느려질 수 있다. JOIN, 사전 일괄 조회, 적절한 캐시를 검토한다.

## 7-3. FOR ALL ENTRIES 빈 테이블

```abap
IF gt_key IS NOT INITIAL.
  SELECT ... FOR ALL ENTRIES IN gt_key ...
ENDIF.
```

원본이 비었을 때 전체 조회 위험이 있으므로 선행 검사가 핵심이다. 중복 키 제거와 필요한 필드 선택도 고려한다.

## 7-4. WHERE 없는 UPDATE/DELETE

실행 전 강사가 학생에게 WHERE 절을 소리 내 읽게 한다. 실무 대량 변경은 대상 건수 사전 조회, 백업/복구 계획, 승인, 로그, 테스트가 필요하다.

## 7-5. 표준 테이블 직접 변경

절대 금지 원칙으로 가르친다.

- SAP 표준 업무 데이터는 여러 테이블·상태·변경문서·후속 프로세스가 연결된다.
- 한 테이블만 직접 UPDATE하면 정합성이 깨질 수 있다.
- 표준 트랜잭션, BAPI, 공개 API, 승인된 확장 지점을 사용한다.
- 이번 과정의 CRUD는 학생별 `ZEDU##_ORDER`에서만 수행한다.

## 7-6. COMMIT 위치

- FORM마다 습관적으로 COMMIT하지 않는다.
- 호출자가 여러 변경을 하나의 업무 단위로 묶어야 할 수 있다.
- 본 과정의 메인 COMMIT은 교육용 단순화다.

## 7-7. 권한 검사 누락

프로그램이 실행된다고 모든 데이터를 보여주거나 바꿔도 되는 것은 아니다. 실제 개발에서는 `AUTHORITY-CHECK`와 조직의 권한 설계를 따른다.

## 7-8. 민감정보를 메시지·로그에 출력

전화번호, 주민번호, 급여, 계좌 등은 디버그 출력·Spool·Application Log에도 남을 수 있다. 최소한만 표시하고 마스킹·보존정책을 따른다.

# 8. 교육 설계의 단순화와 실무 해설

다음 6건은 의도적 설계이므로 본 교재 코드를 “실무형”으로 바꾸지 않는다. 강사는 차이만 설명한다.

| 교육용 단순화 | 왜 이렇게 가르치는가 | 실무에서 확인할 것 |
|---------------|----------------------|---------------------|
| 주문번호 `SELECT MAX + 1` | 채번 흐름을 한눈에 보여줌 | 동시 사용자가 같은 번호를 얻을 수 있으므로 Number Range Object(SNRO)·UUID·DB 전략 검토 |
| 단가 DEC | 통화키 설계를 생략해 계산 집중 | CURR와 참조 통화 WAERS, 통화별 소수 자릿수, 환율·반올림 |
| 금액 미저장 | 수량×단가 계산을 LOOP에서 연습 | 가격 시점 보존·감사·성능·재계산 가능성을 보고 저장 여부 결정 |
| 메인 프로그램에서 COMMIT | CRUD 결과를 즉시 관찰 | LUW 경계, 여러 변경의 원자성, BAPI commit 패턴, Update Task |
| `POPUP_GET_VALUES` | Dynpro 없이 입력 흐름 구현 | 검증·사용성·복잡한 화면은 Dynpro/Fiori/UI 기술과 서비스 계층 검토 |
| 잠금 없는 UPDATE | 3일 분량과 선행지식 보호 | Lock Object의 ENQUEUE/DEQUEUE, 낙관적 잠금, 변경 충돌 처리 |

## 8-1. MAX+1 동시성 이야기

두 사용자가 동시에 MAX=20을 읽으면 둘 다 21을 만들 수 있다. 한 명만 성공하고 다른 한 명은 키 중복이 나거나, 더 복잡한 키에서는 예측하기 어려운 충돌이 생긴다. “지금 데이터가 20건이라 괜찮다”와 “알고리즘이 동시성 안전하다”는 다른 말이다.

## 8-2. 단가와 금액 이야기

실무에서 100이라는 숫자는 통화가 KRW인지 JPY인지 USD인지에 따라 의미와 소수 자릿수가 다르다. 가격결정 시점의 단가를 주문에 스냅샷으로 남길지, 마스터를 매번 다시 읽을지도 업무 결정이다.

## 8-3. 잠금 이야기

두 사용자가 같은 주문을 열고 각각 수량을 바꾸면 나중 저장한 값이 앞 변경을 덮을 수 있다. SAP Lock Object는 업무 객체 단위의 논리 잠금을 제공한다. DB Lock과 완전히 같은 개념으로 단순화하지 않는다.

7. Dump면 ST22, 권한이면 즉시 SU53, Job이면 SM37

| 권한 | SE41 복사 실패, 개발 불가 | SU53과 기준본 전환 |

| 흐름 | 메시지 후 계속 실행 | 타입과 RETURN 확인 |

# 10. 예상 질문 모음

**Q. ABAP은 앞으로도 쓸모가 있나요?**

A. 기존 핵심 업무와 S/4HANA 확장에 여전히 중요하다. 다만 classic 리포트만이 아니라 OO ABAP, CDS, RAP, API, Fiori와 함께 학습해야 선택지가 넓어진다.

**Q. 신문법만 배우면 구문법은 몰라도 되나요?**

A. 신규 코드는 현대 문법을 쓰더라도 기존 운영 코드와 SAP 표준을 읽으려면 구문법 이해가 필요하다. 이 과정은 읽기 기반을 먼저 만든다.

**Q. 모든 것을 Z로 새로 만들면 되나요?**

A. 먼저 표준 기능·설정·공개 API·확장 지점으로 해결 가능한지 본다. 불필요한 커스텀은 업그레이드와 운영 비용을 높인다.

**Q. SE16N에서 데이터를 직접 고쳐도 되나요?**

A. 교육용 Z테이블도 정해진 로더와 프로그램을 우선한다. 운영 표준 데이터 직접 변경은 하지 않는다.

**Q. 성능은 언제부터 신경 써야 하나요?**

A. 요구사항과 예상 데이터량을 설계 때부터 고려한다. 측정 없이 미세 최적화하지 말고 SQL Trace·Runtime Analysis로 병목을 확인한다.

**Q. Dump가 나면 프로그램이 모두 망가진 건가요?**

A. 아니다. 처리되지 않은 예외나 런타임 조건의 기록이다. ST22에서 예외, 소스 위치, 변수, 호출 스택을 읽으면 원인을 좁힐 수 있다.

# 11. 쉬는 시간용 이야기와 동기부여 소재

## 11-1. “오타보다 데이터” 이야기

코드가 완벽해 보여도 테스트 데이터가 다른 클라이언트에 있거나 상태가 예상과 다르면 결과가 0건이다. 좋은 개발자는 소스만 보지 않고 입력·환경·권한·데이터를 함께 본다는 메시지로 연결한다.

## 11-2. “표준 테이블 한 줄” 이야기

판매오더 한 건은 화면상 한 문서지만 저장·상태·가격·일정·변경이력·회계 후속처리가 여러 객체에 걸친다. 한 테이블 직접 수정이 왜 위험한지 설명하는 소재다. 실제 회사 사례를 말할 때 고객명과 수치를 식별 불가능하게 바꾼다.

## 11-3. 디버깅 커리어 이야기

초급자는 코드를 빨리 쓰는 능력을 중요하게 보지만, 운영에서는 낯선 코드와 데이터에서 원인을 좁히고 안전한 수정 범위를 정하는 능력이 더 큰 가치를 만든다. U05를 “에러 찾기 도구”가 아니라 “가설을 검증하는 방법”으로 소개한다.

## 11-4. 다음 학습 로드맵

```text
Classic ABAP 읽기·리포트
→ OO ABAP와 예외·테스트
→ OO ALV/SALV
→ DDIC 심화·CDS View
→ API/BAPI/BAdI/Enhancement
→ OData/Fiori 또는 RAP
→ 성능·권한·Transport·운영
```

학생의 직무가 운영 유지보수인지, 신규 S/4 개발인지, 모듈 컨설팅인지에 따라 다음 순서를 조정한다.
