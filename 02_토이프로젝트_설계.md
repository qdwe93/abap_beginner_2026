## 1. 스토리 (Day 1 OT에서 학생에게 제시)

> "여러분은 작은 회사의 신입 ABAP 개발자입니다. 3일 뒤까지 **주문 관리 프로그램**을 만들어야 합니다.
> 고객과 제품 마스터는 이미 시스템에 있고(선배가 만들어 둠), 여러분은 **주문 테이블을 직접 설계**하고
> 주문을 **조회·등록·수정·삭제**하는 프로그램을 완성합니다."

- 고객/제품 마스터 = 강사가 미리 만들어 둔 **공용 테이블** (전원 공유, 조회만)
- 주문 테이블 = **학생별로 직접 생성** (Day 1 SE11 실습의 결과물)
- 이 구조의 의도: "이미 있는 것을 활용 + 내 것을 만들기" 실무 감각, JOIN 실습 재료 확보, SE11 실습량을 1테이블로 제한해 시간 준수

## 2. 최종 완성 모습 (프로그램 `ZEDU##_MAIN`)

### 화면 흐름

```
[선택화면]                          [ALV 목록 화면]
 주문번호   (SELECT-OPTIONS)         툴바: [등록] [수정] [삭제] + 표준버튼
 고객ID     (SELECT-OPTIONS)   →     주문번호|고객명|제품명|수량|단가|금액|상태|주문일|등록자
 주문일     (SELECT-OPTIONS)         (커서 행 기준으로 수정/삭제 대상 결정)
 상태       (PARAMETERS, 옵션)
```

### 기능 명세

| 기능 | 동작 | 사용 기술 (배운 유닛) |
|------|------|----------------------|
| **조회** | 선택조건으로 주문+고객+제품 JOIN 조회, 금액(수량×단가) 계산, ALV 출력 | U11~U15, 종합실습 ② |
| **등록** | [등록] 버튼 → 입력 팝업(고객ID/제품ID/수량/비고) → 검증 FM 호출 → 주문번호 채번(MAX+1) → INSERT → 성공 메시지 → 목록 갱신 | U16, U17, U18, U19 |
| **수정** | 커서 행 선택 → [수정] → 팝업(수량/상태 변경, 상태는 N↔C만) → UPDATE → 메시지 → 갱신. **완료(C) 주문은 수정 불가(에러), 취소(X) 주문도 수정 불가(리터럴 메시지)** | U17, U18, U19 |
| **삭제** | 커서 행 선택 → [삭제] → 확인 팝업(POPUP_TO_CONFIRM) → DELETE → 메시지 → 갱신. **신규(N) 주문만 삭제 가능(에러)** | U17, U18, U19 |

- 입력 팝업: `POPUP_GET_VALUES` (표준 FM, 화면 개발 없이 팝업 입력 — Dynpro 미학습 상태에서 최적)
- 상태 업무 규칙(수정·삭제 제한)은 MESSAGE E 실습을 자연스럽게 유도하기 위한 장치
- **취소(X)는 선택 구현**: 도메인 고정값·샘플 데이터에는 X가 존재(조회/필터 연습용)하지만, 취소 처리 기능(N→X)은 기본 과정에 넣지 않고 확장 과제로 분리

## 3. 데이터 모델

```
ZEDU_CUST (고객, 공용)      ZEDU##_ORDER (주문, 학생별)      ZEDU_PROD (제품, 공용)
 CUST_ID ◄──────────────────  CUST_ID                          
 CUST_NAME                    PROD_ID  ──────────────────────► PROD_ID
 CITY                         QUANTITY                          PROD_NAME
 PHONE                        STATUS                            PRICE (단가)
                              ORDER_DATE                        
                              REMARK / ERNAM                    금액 = QUANTITY × PRICE (저장 안 함, 조회 시 계산)
```

### 3-1. 공용 마스터 (강사 사전 생성)

**ZEDU_CUST — 고객 마스터**

| 필드 | 키 | 타입 | 설명 |
|------|:--:|------|------|
| MANDT | ✔ | CLNT | 클라이언트 |
| CUST_ID | ✔ | CHAR 6 (DE `ZEDU_CUSTID`) | 고객 ID (예: C00001) |
| CUST_NAME | | CHAR 30 (DE `ZEDU_CUSTNAME`) | 고객명 |
| CITY | | CHAR 20 | 도시 |
| PHONE | | CHAR 20 | 전화번호 |

**ZEDU_PROD — 제품 마스터**

| 필드 | 키 | 타입 | 설명 |
|------|:--:|------|------|
| MANDT | ✔ | CLNT | 클라이언트 |
| PROD_ID | ✔ | CHAR 8 (DE `ZEDU_PRODID`) | 제품 ID (예: P0000001) |
| PROD_NAME | | CHAR 30 (DE `ZEDU_PRODNAME`) | 제품명 |
| PRICE | | DEC 11,2 (DE `ZEDU_PRICE`) | 단가 (통화 단순화를 위해 CURR 대신 DEC — 강사 가이드에 실무와의 차이 설명 수록) |

**공용 데이터 엘리먼트 9종 (강사 사전 생성)** — 학생 테이블·스트럭처에서 재사용:
`ZEDU_CUSTID`(CHAR6, 고객 ID), `ZEDU_PRODID`(CHAR8, 제품 ID), `ZEDU_CUSTNAME`(CHAR30, 고객명), `ZEDU_PRODNAME`(CHAR30, 제품명), `ZEDU_PRICE`(DEC 11,2, 단가), `ZEDU_AMOUNT`(DEC 15,2, 금액), `ZEDU_QTY`(INT4, 주문수량), `ZEDU_ORDDATE`(DATS, 주문일), `ZEDU_REMARK`(CHAR40, 비고)

> ALV 컬럼 제목은 데이터 엘리먼트의 필드 라벨에서 자동으로 나온다. 그래서 ALV에 표시되는 스트럭처(ZEDU##_S_LIST)의 모든 필드는 DE를 참조하도록 설계 (라벨 없는 빌트인 타입이면 컬럼 제목이 비게 됨)

**확장 과제 (빠른 학생용, TP-5 및 수시 투입)**

1. 더블클릭 시 해당 주문 상세를 팝업으로 표시
2. 상태별 행 색상 (취소=빨강, 완료=초록)
3. ALV 하단 금액 합계 행
4. 체크박스 다중 선택으로 여러 건 삭제
5. **[주문 취소] 버튼 추가 (N → X 상태 변경)** — 기본 과정에서 뺀 취소 처리 기능. 수정 팝업에서 X 입력을 허용하는 방식으로 구현해도 됨
6. 조회 로직을 신문법(new Open SQL + 인라인 선언)으로 리팩터링

## 7. 설계상 단순화 결정 (강사 가이드에 "실무에서는" 보충 예정)

| 단순화 | 실무에서는 |
|--------|-----------|
| 주문번호 채번 = SELECT MAX + 1 | Number Range Object (SNRO) 사용 |
| 단가 DEC (통화 없음) | CURR + 참조 통화 필드(WAERS) |
| 금액 미저장(조회 시 계산) | 요구사항에 따라 저장 여부 결정 |
| COMMIT WORK를 메인 프로그램에서 | LUW 설계, BAPI 패턴(BAPI_TRANSACTION_COMMIT) |
| 팝업 = POPUP_GET_VALUES | 커스텀 Dynpro / Module Pool |
| 잠금 없이 UPDATE | ENQUEUE/DEQUEUE 잠금 오브젝트 |
