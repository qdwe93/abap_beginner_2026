## U09. 내부 테이블 ① 선언·APPEND·LOOP (09:15~10:05)

### 🎯 목표
- 라인 타입 / 워크에어리어 / 내부 테이블의 관계를 이해한다
- APPEND로 행을 쌓고 LOOP로 꺼낼 수 있다

### 📖 개념

**내부 테이블** = 프로그램 **메모리 안의** 엑셀 시트. DB 테이블(어제 만든 것, 디스크에 영구 저장)과 달리 프로그램이 끝나면 사라집니다. ABAP 프로그래밍의 90%는 "DB에서 읽어와 → 내부 테이블에서 가공 → 출력"입니다.

```
TYPES ty_order   : 행의 설계도 (라인 타입)
DATA  gs_order   : 행 1개짜리 변수 (워크에어리어, gs_)
DATA  gt_order   : 행 여러 개 (내부 테이블, gt_)

[gt_order]                          [gs_order]
 1 | P0000003 | 10 | N     ←APPEND─  P0000004 | 30 | X
 2 | P0000001 |  2 | C
```

- `APPEND gs TO gt.` — 워크에어리어 내용을 테이블 **맨 뒤에 복사**
- `LOOP AT gt INTO gs. ... ENDLOOP.` — 한 행씩 워크에어리어로 꺼내며 반복. 반복 중 `sy-tabix` = 현재 행 번호

### 💻 따라치기 — `ZEDU##_U09`

```abap
REPORT zedu##_u09.

* 1) 라인 타입 정의 (어제 만든 DE를 타입으로 재사용!)
TYPES: BEGIN OF ty_order,
         order_id TYPE zedu##_ordid,
         prod_id  TYPE zedu_prodid,
         quantity TYPE zedu_qty,
         status   TYPE zedu##_status,
       END OF ty_order.

* 2) 내부 테이블 + 워크에어리어
DATA: gt_order TYPE TABLE OF ty_order,
      gs_order TYPE ty_order.

DATA: gv_lines TYPE i.

* 3) 행 채우기 -> APPEND
gs_order-order_id = 1.
gs_order-prod_id  = 'P0000003'.
gs_order-quantity = 10.
gs_order-status   = 'N'.
APPEND gs_order TO gt_order.

CLEAR gs_order.                 " ★ 잔여값 사고 방지. APPEND 전 CLEAR 습관!
gs_order-order_id = 2.
gs_order-prod_id  = 'P0000001'.
gs_order-quantity = 2.
gs_order-status   = 'C'.
APPEND gs_order TO gt_order.

CLEAR gs_order.
gs_order-order_id = 3.
gs_order-prod_id  = 'P0000004'.
gs_order-quantity = 30.
gs_order-status   = 'X'.
APPEND gs_order TO gt_order.

* 4) 전체 읽기
LOOP AT gt_order INTO gs_order.
  WRITE: / sy-tabix, gs_order-order_id, gs_order-prod_id,
           gs_order-quantity, gs_order-status.
ENDLOOP.

* 5) 조건 걸고 읽기
WRITE / '--- 신규(N)만 ---'.
LOOP AT gt_order INTO gs_order WHERE status = 'N'.
  WRITE: / gs_order-order_id, gs_order-prod_id.
ENDLOOP.

* 6) 건수
DESCRIBE TABLE gt_order LINES gv_lines.
WRITE: / '총', gv_lines, '건'.
```

> 💡 3)에서 CLEAR를 하나 지우고 실행해보세요 — 이전 행 값이 남아 섞입니다. 실무에서 아주 흔한 버그입니다.

> ✨ **모던 ABAP으로는 이렇게**
> ```abap
> TYPES ty_order_tab TYPE TABLE OF ty_order.
>
> DATA(lt_order) = VALUE ty_order_tab(
>   ( order_id = 1  prod_id = 'P0000003'  quantity = 10  status = 'N' )
>   ( order_id = 2  prod_id = 'P0000001'  quantity = 2   status = 'C' ) ).
>
> LOOP AT lt_order INTO DATA(ls_order).         " 워크에어리어 인라인 선언
>   WRITE: / ls_order-order_id.
> ENDLOOP.
> ```
> VALUE #( ... )로 선언과 데이터 구성이 한 문장으로 끝납니다.

## U10. 내부 테이블 ② READ·SORT·MODIFY·DELETE (10:20~11:10)

### 🎯 목표
- READ TABLE로 특정 행을 찾고 sy-subrc / sy-tabix를 해석할 수 있다
- SORT / MODIFY / DELETE / CLEAR로 내부 테이블을 조작할 수 있다

### 📖 개념

| 문장 | 역할 |
|------|------|
| `READ TABLE gt INTO gs WITH KEY f = 값.` | 조건으로 **한 행** 찾기 (첫 매칭) |
| `READ TABLE gt INTO gs INDEX n.` | n번째 행 읽기 |
| `SORT gt BY f1 [DESCENDING] f2.` | 정렬 |
| `MODIFY gt FROM gs INDEX n.` | n번째 행을 gs 내용으로 교체 |
| `DELETE gt WHERE 조건.` / `DELETE gt INDEX n.` | 행 삭제 |
| `CLEAR gt.` | 전부 비우기 (옛 코드의 `REFRESH gt.`와 동일) |

**sy-subrc** = "직전 명령이 성공했나?" — 0이면 성공. READ/SELECT 등 수많은 명령이 결과를 여기 남깁니다. **READ 후에는 반드시 sy-subrc 확인** — 오늘부터 몸에 새길 습관입니다.

### 💻 따라치기 — `ZEDU##_U10`

(1)~(3) 데이터 만들기는 U09와 동일 — `ZEDU##_U09`에서 TYPES~APPEND 부분(`DATA gv_lines TYPE i.` 선언 포함)을 복사해 오세요. 그 아래에:

```abap
* 1) READ TABLE: 조건으로 한 행 찾기
READ TABLE gt_order INTO gs_order WITH KEY order_id = 2.
IF sy-subrc = 0.
  WRITE: / '찾음! 행번호:', sy-tabix, '/ 제품:', gs_order-prod_id.
ELSE.
  WRITE / '해당 주문 없음'.
ENDIF.

READ TABLE gt_order INTO gs_order WITH KEY order_id = 99.
WRITE: / 'order_id=99 조회 sy-subrc =', sy-subrc.   " 4 (실패)

* 2) SORT: 수량 많은 순
SORT gt_order BY quantity DESCENDING.
WRITE / '--- 수량 내림차순 ---'.
LOOP AT gt_order INTO gs_order.
  WRITE: / gs_order-order_id, gs_order-quantity.
ENDLOOP.

* 3) MODIFY: 2번 주문 수량을 99로
READ TABLE gt_order INTO gs_order WITH KEY order_id = 2.
IF sy-subrc = 0.
  gs_order-quantity = 99.
  MODIFY gt_order FROM gs_order INDEX sy-tabix.   " READ가 남긴 행번호 재사용
ENDIF.

* 4) DELETE: 취소 주문 제거
DELETE gt_order WHERE status = 'X'.

WRITE / '--- 수정/삭제 후 ---'.
LOOP AT gt_order INTO gs_order.
  WRITE: / gs_order-order_id, gs_order-quantity, gs_order-status.
ENDLOOP.

* 5) 전부 비우기
CLEAR gt_order.
DESCRIBE TABLE gt_order LINES gv_lines.
WRITE: / 'CLEAR 후 건수:', gv_lines.     " 0
```

> ⚠️ READ 직후 다른 명령을 실행하면 sy-tabix / sy-subrc가 **덮어써집니다**. 쓰려면 바로 쓰거나 변수에 보관하세요.

> ✨ **모던 ABAP으로는 이렇게**
> ```abap
> IF line_exists( gt_order[ order_id = 2 ] ).        " 존재 확인
>   DATA(ls_found) = gt_order[ order_id = 2 ].       " 테이블 표현식으로 행 읽기
>   WRITE / lines( gt_order ).                       " 건수
> ENDIF.
> ```
> `gt[ ... ]`는 없는 행이면 **예외(덤프)** — 그래서 line_exists와 짝으로 씁니다.

## U11. Open SQL ① SELECT 기초 (11:20~12:00 개념·따라치기 / 13:00~13:20 퀴즈)

### 🎯 목표
- SELECT SINGLE / SELECT ... INTO TABLE로 DB를 조회할 수 있다
- WHERE로 조건을 걸고 sy-subrc / sy-dbcnt를 확인할 수 있다

### 📖 개념

**Open SQL** = ABAP 안에서 쓰는 SQL. DB 종류(HANA, Oracle...)와 무관하게 동일하게 동작합니다.

```abap
SELECT SINGLE * FROM 테이블 INTO 워크에어리어 WHERE 키 = 값.   " 딱 1건
SELECT *        FROM 테이블 INTO TABLE 내부테이블 WHERE 조건.  " 여러 건 한번에
```

- 어제 배운 내부 테이블이 **SELECT의 그릇**이 됩니다
- `sy-subrc` = 0 조회 성공 / 4 결과 없음, `sy-dbcnt` = 읽은 건수
- `SELECT SINGLE`은 **키 전체를 지정**해 1건을 특정할 때 쓰는 것이 정석

### 💻 따라치기 — `ZEDU##_U11`

```abap
REPORT zedu##_u11.

DATA: gs_prod  TYPE zedu_prod,
      gt_prod  TYPE TABLE OF zedu_prod,
      gt_order TYPE TABLE OF zedu##_order,
      gs_order TYPE zedu##_order.

* 1) SELECT SINGLE: 키로 1건
SELECT SINGLE * FROM zedu_prod INTO gs_prod
  WHERE prod_id = 'P0000001'.

IF sy-subrc = 0.
  WRITE: / gs_prod-prod_id, gs_prod-prod_name, gs_prod-price.
ELSE.
  WRITE / '제품이 없습니다.'.
ENDIF.

* 2) 여러 건: INTO TABLE
SELECT * FROM zedu_prod INTO TABLE gt_prod
  WHERE price >= 100000.

WRITE: / '10만원 이상 제품:', sy-dbcnt, '건'.
LOOP AT gt_prod INTO gs_prod.
  WRITE: / gs_prod-prod_id, gs_prod-prod_name, gs_prod-price.
ENDLOOP.

* 3) 내 주문 테이블 조회
SELECT * FROM zedu##_order INTO TABLE gt_order
  WHERE status = 'N'.

WRITE: / '내 신규 주문:', sy-dbcnt, '건'.
LOOP AT gt_order INTO gs_order.
  WRITE: / gs_order-order_id, gs_order-prod_id, gs_order-quantity.
ENDLOOP.
```

> 📦 **옛 코드에서 만나게 될 것**: `SELECT * FROM ... . ... ENDSELECT.` — 행 단위 처리 구조여서 집합 처리 대비 불리하고 코드도 장황해지므로, 신규 코드는 `INTO TABLE` + `LOOP`를 우선합니다. (읽을 줄만 알면 됩니다)

## U12. Open SQL ② JOIN과 집계 (13:20~14:10)

### 🎯 목표
- INNER JOIN으로 여러 테이블을 한 번에 조회할 수 있다
- COUNT/SUM, GROUP BY, ORDER BY, UP TO n ROWS를 쓸 수 있다

### 📖 개념

주문 테이블에는 고객 "ID"만 있습니다. 고객 "이름"은 ZEDU_CUST에 있죠. 두 테이블을 붙여서 한 번에 가져오는 것이 **JOIN**:

```abap
FROM 주문 AS a
INNER JOIN 고객 AS b ON a~cust_id = b~cust_id   " a~ : 별칭~필드
```

INNER JOIN = **양쪽 다 있는 것만** 결과에 나옴. (한쪽만 있어도 나오게 하려면 LEFT OUTER JOIN — 소개만)

### 💻 따라치기 — `ZEDU##_U12`

```abap
REPORT zedu##_u12.

TYPES: BEGIN OF ty_list,
         order_id  TYPE zedu##_ordid,
         cust_name TYPE zedu_custname,
         prod_name TYPE zedu_prodname,
         quantity  TYPE zedu_qty,
         price     TYPE zedu_price,
       END OF ty_list.

TYPES: BEGIN OF ty_stat,
         status TYPE zedu##_status,
         cnt    TYPE i,
       END OF ty_stat.

DATA: gt_list  TYPE TABLE OF ty_list,
      gs_list  TYPE ty_list,
      gt_stat  TYPE TABLE OF ty_stat,
      gs_stat  TYPE ty_stat,
      gt_top   TYPE TABLE OF zedu##_order,
      gs_top   TYPE zedu##_order,
      gv_cnt   TYPE i.

* 1) 3개 테이블 INNER JOIN: 주문 + 고객명 + 제품명/단가
SELECT a~order_id b~cust_name c~prod_name a~quantity c~price
  FROM zedu##_order AS a
  INNER JOIN zedu_cust AS b ON a~cust_id = b~cust_id
  INNER JOIN zedu_prod AS c ON a~prod_id = c~prod_id
  INTO CORRESPONDING FIELDS OF TABLE gt_list
  WHERE a~status = 'N'.

WRITE / '--- 신규 주문 (이름 포함) ---'.
LOOP AT gt_list INTO gs_list.
  WRITE: / gs_list-order_id, gs_list-cust_name, gs_list-prod_name,
           gs_list-quantity, gs_list-price.
ENDLOOP.

* 2) 집계: 건수
SELECT COUNT( * ) FROM zedu##_order INTO gv_cnt.
WRITE: / '전체 주문:', gv_cnt, '건'.

* 3) GROUP BY: 상태별 건수
SELECT status COUNT( * )
  FROM zedu##_order
  INTO TABLE gt_stat
  GROUP BY status.

WRITE / '--- 상태별 건수 ---'.
LOOP AT gt_stat INTO gs_stat.
  WRITE: / gs_stat-status, gs_stat-cnt.
ENDLOOP.

* 4) 정렬 + 상위 3건
SELECT * FROM zedu##_order
  INTO TABLE gt_top
  UP TO 3 ROWS
  ORDER BY quantity DESCENDING.

WRITE / '--- 수량 TOP 3 ---'.
LOOP AT gt_top INTO gs_top.
  WRITE: / gs_top-order_id, gs_top-quantity.
ENDLOOP.
```

> 💡 `INTO CORRESPONDING FIELDS OF TABLE` = **필드 이름이 같은 곳에** 넣기. 이름만 맞으면 순서가 달라도 안전합니다. (이름이 다르면? 그 필드는 비어 있게 됨 — 오타 주의)

> 📦 **FOR ALL ENTRIES**: 실무 코드에서 JOIN만큼 자주 보는 "내부 테이블과 조인하기" 패턴. **원본 테이블이 비어 있으면 전체 조회가 터지는** 유명한 함정이 있습니다 — 강사와 함께 소개만 하고 넘어갑니다. (신문법 세계에서는 JOIN/CDS로 대체 추세)

> ✨ **모던 ABAP으로는 이렇게**
> ```abap
> SELECT a~order_id, b~cust_name, c~prod_name, a~quantity, c~price
>   FROM zedu##_order AS a
>   INNER JOIN zedu_cust AS b ON a~cust_id = b~cust_id
>   INNER JOIN zedu_prod AS c ON a~prod_id = c~prod_id
>   WHERE a~status = 'N'
>   INTO TABLE @DATA(lt_list).
> ```
> JOIN 구문 자체는 같고, 콤마/@/인라인 선언만 얹으면 됩니다.

## U13. 선택화면 PARAMETERS / SELECT-OPTIONS (14:20~15:10)

### 🎯 목표
- PARAMETERS와 SELECT-OPTIONS로 선택화면을 만들 수 있다
- IN 연산자로 범위 조건을 SELECT에 반영할 수 있다

### 📖 개념

지금까지는 조건이 코드에 박혀 있었습니다(`WHERE status = 'N'`). 조건을 **사용자가 실행할 때 입력**하게 만드는 것이 선택화면:

| 문장 | 화면 모습 | 용도 |
|------|-----------|------|
| `PARAMETERS p_x TYPE 타입.` | 입력칸 1개 | 단일 값 |
| `SELECT-OPTIONS s_x FOR 참조필드.` | 입력칸 2개(From~To) + 복수선택 버튼 | 범위·복수 값 |

- SELECT-OPTIONS의 정체: SIGN/OPTION/LOW/HIGH 4개 필드를 가진 **작은 내부 테이블**. `WHERE 필드 IN s_x`가 그걸 해석합니다
- 선택화면 **라벨**은 텍스트 심볼: 메뉴 Goto → Text Elements → **Selection Texts**

### 💻 따라치기 — `ZEDU##_U13`

```abap
REPORT zedu##_u13.

DATA: gs_order TYPE zedu##_order,
      gt_order TYPE TABLE OF zedu##_order,
      gv_cnt   TYPE i.

* 선택화면 (SELECT-OPTIONS는 기존 필드를 참조해 타입을 가져옴)
PARAMETERS: p_stat TYPE zedu##_status.
SELECT-OPTIONS: s_ordid FOR gs_order-order_id,
                s_date  FOR gs_order-order_date.

START-OF-SELECTION.    " (정체는 다음 유닛에서!)

  SELECT * FROM zedu##_order INTO TABLE gt_order
    WHERE order_id   IN s_ordid
      AND order_date IN s_date.

  " 상태는 입력된 경우에만 필터 (PARAMETERS는 IN을 못 쓰므로)
  IF p_stat IS NOT INITIAL.
    DELETE gt_order WHERE status <> p_stat.
  ENDIF.

  DESCRIBE TABLE gt_order LINES gv_cnt.
  WRITE: / '조회 건수:', gv_cnt.
  LOOP AT gt_order INTO gs_order.
    WRITE: / gs_order-order_id, gs_order-cust_id, gs_order-prod_id,
```

**실행하며 관찰할 것**:
1. 조건을 다 비우고 실행 → 20건 전부 (빈 SELECT-OPTIONS = 조건 없음)
2. 주문번호 From 1 To 5 → 구간 조회

## U14. 리포트 이벤트 흐름 (15:25~16:05)

### 🎯 목표
- 리포트 프로그램의 이벤트 실행 순서를 이해한다
- INITIALIZATION에서 기본값을, AT SELECTION-SCREEN에서 검증을 구현할 수 있다

### 📖 개념

리포트 프로그램은 위에서 아래로 그냥 흐르지 않습니다. **정해진 시점에 정해진 이벤트 블록**이 실행됩니다:

```
프로그램 시작
  → INITIALIZATION        (선택화면 나오기 전 1회: 기본값 세팅)
  → [선택화면 표시]
  → AT SELECTION-SCREEN   (실행(F8) 누를 때마다: 입력값 검증)
      └ 검증 실패(MESSAGE E) → 선택화면으로 되돌아감!
  → START-OF-SELECTION    (본 처리: 조회/가공)
  → END-OF-SELECTION      (본 처리 후 마무리)
  → [결과 목록 표시]
```

이벤트 블록은 코드에 적힌 **순서와 무관하게** 정해진 시점에 불립니다. `START-OF-SELECTION.`이라고 쓰는 순간부터가 그 이벤트의 코드입니다.

### 💻 따라치기 — `ZEDU##_U14`

```abap
      gv_check TYPE zedu_custid,

PARAMETERS     p_cust TYPE zedu_custid.

INITIALIZATION.
  MESSAGE '기본 조회기간을 세팅했습니다 (최근 90일)' TYPE 'S'.
  CLEAR s_date.
  s_date-sign   = 'I'.            " Include
  s_date-option = 'BT'.           " BeTween
  s_date-low    = sy-datum - 90.
  s_date-high   = sy-datum.
  APPEND s_date.                  " SELECT-OPTIONS도 내부 테이블이라 APPEND!

* [2] 실행 누를 때마다: 검증
AT SELECTION-SCREEN.
  IF p_cust IS NOT INITIAL.
    SELECT SINGLE cust_id FROM zedu_cust INTO gv_check
      WHERE cust_id = p_cust.

  WRITE / '--- 처리 완료 (END-OF-SELECTION) ---'.
```

## U15. ALV 출력 (16:15~17:15)

### 🎯 목표
- (준비) 서브루틴 FORM/PERFORM으로 코드를 정리할 수 있다
- REUSE_ALV_GRID_DISPLAY로 내부 테이블을 ALV 그리드에 뿌릴 수 있다

### 📖 개념 1 — 서브루틴 (5분 선행 학습)

코드가 길어지기 시작합니다. **FORM**(서브루틴)으로 로직에 이름을 붙여 상자에 담습니다:

```abap
START-OF-SELECTION.
  PERFORM get_data.     " 호출

FORM get_data.          " 정의
  ...
ENDFORM.
```

이걸 지금 배우는 이유: 잠시 후 ALV가 **"이 이름의 FORM을 호출해 줘"** 방식(콜백)으로 동작하기 때문입니다. (파라미터 넘기기 USING/CHANGING은 내일 필요할 때 소개)

### 📖 개념 2 — ALV

WRITE 목록은 초라합니다. **ALV(ABAP List Viewer)** = SAP 표준 목록 출력 도구. 정렬·필터·합계·엑셀 다운로드가 **공짜**로 따라옵니다. 고전 리포트의 사실상 표준 출력이며, 표준 함수 `REUSE_ALV_GRID_DISPLAY`를 호출해 씁니다.

(함수 호출 문법은 내일 정식으로 배웁니다. 오늘은 "표준 기능을 빌려 쓰는 정해진 서식" 정도로 따라 치세요)

### 💻 따라치기 — `ZEDU##_U15`

```abap
REPORT zedu##_u15.

TYPE-POOLS: slis.                 " ALV 관련 타입 꾸러미 (관례적 선언)

DATA: gt_list TYPE TABLE OF zedu##_s_list,   " 어제 만든 스트럭처가 드디어 등장!
      gs_list TYPE zedu##_s_list.

START-OF-SELECTION.
  PERFORM get_data.
  PERFORM show_alv.

*&---------------------------------------------------------------------*
*& 조회: 주문 + 고객명 + 제품명/단가 (U12의 JOIN)
*&---------------------------------------------------------------------*
FORM get_data.
  SELECT a~order_id a~cust_id b~cust_name
         a~prod_id  c~prod_name c~price
         a~quantity a~status a~order_date a~remark a~ernam
    FROM zedu##_order AS a
    INNER JOIN zedu_cust AS b ON a~cust_id = b~cust_id
    INNER JOIN zedu_prod AS c ON a~prod_id = c~prod_id
    INTO CORRESPONDING FIELDS OF TABLE gt_list.

  " 금액 = 수량 x 단가 (DB에 없는 계산 필드 채우기)
  LOOP AT gt_list INTO gs_list.
    gs_list-amount = gs_list-quantity * gs_list-price.
    MODIFY gt_list FROM gs_list TRANSPORTING amount.
  ENDLOOP.
ENDFORM.

*&---------------------------------------------------------------------*
*& ALV 출력
*&---------------------------------------------------------------------*
FORM show_alv.
  DATA ls_layout TYPE slis_layout_alv.

  ls_layout-zebra             = 'X'.   " 줄무늬
  ls_layout-colwidth_optimize = 'X'.   " 컬럼폭 자동 조절

  CALL FUNCTION 'REUSE_ALV_GRID_DISPLAY'
    EXPORTING
      i_callback_program = sy-repid            " 나(현재 프로그램)를 알려줌
      i_structure_name   = 'ZEDU##_S_LIST'     " ★ DDIC 스트럭처로 컬럼 자동 구성
      is_layout          = ls_layout
      i_grid_title       = '주문 목록'
    TABLES
      t_outtab           = gt_list             " 출력할 내부 테이블
    EXCEPTIONS
      program_error      = 1
      OTHERS             = 2.
  IF sy-subrc <> 0.
    WRITE / 'ALV 호출 오류'.
  ENDIF.
ENDFORM.
```

**실행 후 관찰** 📷:
1. **컬럼 제목이 저절로 한글로!** — 어제 데이터 엘리먼트에 넣은 라벨입니다. DDIC에 정의를 쌓으면 화면이 공짜로 따라오는 것, 이게 SAP 방식입니다
2. 툴바 기능 직접 눌러보기: 정렬 ▲▼, 필터, **합계(Σ) — AMOUNT 컬럼에**, 엑셀 내보내기
3. `i_structure_name`을 지우고 실행하면? → 오류. 컬럼 구성 정보가 없기 때문 (수동 구성법 = 필드 카탈로그, 추가 퀴즈집에서)

> ✨ **모던 ABAP으로는 이렇게 (객체지향 ALV)**
> ```abap
> DATA lo_alv TYPE REF TO cl_salv_table.
>
> TRY.
>     cl_salv_table=>factory( IMPORTING r_salv_table = lo_alv
>                             CHANGING  t_table      = gt_list ).
>     lo_alv->display( ).
>   CATCH cx_salv_msg INTO DATA(lx_salv).
>     DATA(lv_salv_text) = lx_salv->get_text( ).
>     MESSAGE lv_salv_text TYPE 'S' DISPLAY LIKE 'E'.
> ENDTRY.
> ```
> 클래스 기반 SALV — **핵심 호출부는 세 줄**이고 필드 카탈로그가 필요 없습니다. `FACTORY`의 예외 처리를 빼도 시스템에 따라 경고 후 활성화될 수 있지만, 실행 시 런타임 오류 위험이 있으므로 완결 예제처럼 `TRY ... CATCH`로 처리합니다. 객체지향 ABAP을 배우면 첫 실전 상대가 됩니다.
