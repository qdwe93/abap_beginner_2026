# SAP ABAP 3일 기초교육 — 추가 퀴즈집 (STEP 4)

> 대상 시스템: S/4HANA 2023 온프레미스 / Classic ABAP 중심
> 소재: 본 과정의 간이 주문 관리 toy project

## 사용법

- 각 유닛에는 **쉬운 변형(E)** 1개와 **심화 변형(D)** 1개가 있습니다.
- 쉬운 변형은 본 실습을 한 번 더 연습하는 문제입니다. 진도가 느린 학생에게 먼저 제공합니다.
- 심화 변형은 해당 유닛까지 배운 내용을 조합하는 문제입니다. 빨리 끝난 학생에게 제공합니다.
- 코딩 문제는 실행 가능한 선언부와 흐름을 제공하고, 학생은 `TODO`만 작성합니다.
- `##`는 학생 번호 두 자리 자리표시자입니다. 자기 번호로 바꿔 사용합니다.
- 정답은 여러 방법 중 한 가지 예시입니다. 먼저 실행 결과를 확인한 뒤 해설을 읽습니다.

# Day 1 — 문법 기초와 DDIC

## U01. SE38 첫 프로그램과 WRITE

### E01 — 주문 접수 안내문

다음 스켈레톤의 TODO를 채워 세 줄을 출력하세요.

```abap
REPORT zedu##_u01e.

* TODO: WRITE를 세 번 사용해 아래 문장을 한 줄씩 출력하세요.
* 간이 주문 관리
* 오늘부터 개발 시작
* 학생 번호: ##
```

**정답**

```abap
WRITE / '간이 주문 관리'.
WRITE / '오늘부터 개발 시작'.
WRITE / '학생 번호: ##'.
```

**해설**: `/`는 새 줄에서 출력을 시작합니다. 이 단계에서는 문자열을 변수에 담지 않고 `WRITE`의 기본 형태와 저장·활성화·실행 흐름을 반복합니다.

### D01 — 주문 요약 한 줄 만들기

```abap
REPORT zedu##_u01d.

* TODO: 한 WRITE 문장으로 다음 값을 같은 줄에 출력하세요.
* 주문번호: 0000000001 / 상태: N
```

**정답**

```abap
WRITE: '주문번호:', '0000000001', '/', '상태:', 'N'.
```

**해설**: 콜론 뒤에 출력 항목을 쉼표로 이어 쓸 수 있습니다. 여기의 `/`는 따옴표 안의 문자이므로 줄바꿈 명령이 아닙니다.

## U02. 변수와 데이터 타입

### E02 — 주문 금액 계산

```abap
REPORT zedu##_u02e.

DATA: gv_qty    TYPE i,
      gv_price  TYPE p LENGTH 8 DECIMALS 2,
      gv_amount TYPE p LENGTH 10 DECIMALS 2.

gv_qty   = 4.
gv_price = '350000.00'.

* TODO 1: 수량과 단가로 금액을 계산하세요.
* TODO 2: 수량, 단가, 금액을 출력하세요.
```

**정답**

```abap
gv_amount = gv_qty * gv_price.
WRITE: / '수량:', gv_qty,
       / '단가:', gv_price,
       / '금액:', gv_amount.
```

**해설**: 금액은 수량×단가이며 예상 결과는 1,400,000.00입니다. 소수 자릿수가 필요한 금액은 `TYPE p ... DECIMALS 2`로 선언합니다.

### D02 — 할인 후 결제금액

```abap
REPORT zedu##_u02d.

DATA: gv_qty      TYPE i VALUE 10,
      gv_price    TYPE p LENGTH 8 DECIMALS 2 VALUE '45000.00',
      gv_amount   TYPE p LENGTH 10 DECIMALS 2,
      gv_discount TYPE p LENGTH 10 DECIMALS 2,
      gv_payment  TYPE p LENGTH 10 DECIMALS 2.

CONSTANTS gc_rate TYPE p LENGTH 3 DECIMALS 2 VALUE '0.05'.

* TODO 1: 총액, 5% 할인액, 결제금액을 계산하세요.
* TODO 2: 세 값을 출력하세요.
```

**정답**

```abap
gv_amount   = gv_qty * gv_price.
gv_discount = gv_amount * gc_rate.
gv_payment  = gv_amount - gv_discount.

WRITE: / '총액:', gv_amount,
       / '할인:', gv_discount,
       / '결제:', gv_payment.
```

**해설**: 반복해서 바뀌지 않는 할인율은 상수로 둡니다. 예상 결제금액은 427,500.00입니다.

## U03. 조건문 IF / CASE

### E03 — 상태명 바꾸기

```abap
REPORT zedu##_u03e.

DATA: gv_status TYPE c LENGTH 1 VALUE 'X',
      gv_text   TYPE c LENGTH 10.

* TODO: CASE로 N=신규, C=완료, X=취소, 그 외=알수없음을 넣으세요.

WRITE: / gv_status, gv_text.
```

**정답**

```abap
CASE gv_status.
  WHEN 'N'.
    gv_text = '신규'.
  WHEN 'C'.
    gv_text = '완료'.
  WHEN 'X'.
    gv_text = '취소'.
  WHEN OTHERS.
    gv_text = '알수없음'.
ENDCASE.
```

**해설**: 하나의 상태값을 여러 고정값과 비교하므로 `CASE`가 읽기 쉽습니다. `WHEN OTHERS`는 예상하지 못한 값도 빠짐없이 처리합니다.

### D03 — 주문 우선순위 판정

```abap
REPORT zedu##_u03d.

DATA: gv_qty      TYPE i VALUE 25,
      gv_status   TYPE c LENGTH 1 VALUE 'N',
      gv_priority TYPE c LENGTH 10.

* TODO: 아래 규칙대로 IF를 작성하세요.
* 취소(X)                 -> 처리제외
* 신규(N)이면서 수량 20 이상 -> 긴급
* 신규(N)                 -> 일반
* 그 외                   -> 완료

WRITE: / '우선순위:', gv_priority.
```

**정답**

```abap
IF gv_status = 'X'.
  gv_priority = '처리제외'.
ELSEIF gv_status = 'N' AND gv_qty >= 20.
  gv_priority = '긴급'.
ELSEIF gv_status = 'N'.
  gv_priority = '일반'.
ELSE.
  gv_priority = '완료'.
ENDIF.
```

**해설**: 더 구체적인 조건을 먼저 둡니다. `N`을 먼저 검사하면 수량 20 이상인 신규 주문도 모두 일반으로 분류됩니다.

### D04 — 짝수 수량의 합

```abap
REPORT zedu##_u04d.

DATA gv_sum TYPE i.

* TODO: 1부터 20까지 중 짝수만 더해 gv_sum에 넣으세요.
```

## U05. 디버거 기초

### E05 — 사라지는 합계 찾기

아래 프로그램은 1~5의 합을 구하려 하지만 결과가 5입니다. `/h` 또는 중단점을 사용해 `gv_sum`이 언제 잘못되는지 찾으세요.

```abap
REPORT zedu##_u05e.

DATA gv_sum TYPE i.

DO 5 TIMES.
  gv_sum = 0.                       " BUG
  gv_sum = gv_sum + sy-index.
ENDDO.

WRITE / gv_sum.
```

**정답**: `gv_sum = 0.`을 반복문 밖으로 이동하거나 삭제합니다.

```abap
gv_sum = 0.
DO 5 TIMES.
  gv_sum = gv_sum + sy-index.
ENDDO.
```

**해설**: 중단점을 루프 안에 두고 F5/F6로 실행하면 매 회차 합계가 0으로 초기화되는 순간을 볼 수 있습니다. 수정 후 결과는 15입니다.

### D05 — 잘못된 상태 분기 추적

```abap
REPORT zedu##_u05d.

DATA: gv_status TYPE c LENGTH 1 VALUE 'N',
      gv_text   TYPE c LENGTH 10.

IF gv_status = 'C'.
  gv_text = '신규'.                 " BUG
ELSEIF gv_status = 'N'.
  gv_text = '완료'.                 " BUG
ELSE.
  gv_text = '기타'.
ENDIF.

WRITE / gv_text.
```

**과제**: 조건식과 대입 결과를 디버거로 관찰한 뒤 두 줄을 고치세요.

**정답**

```abap
IF gv_status = 'C'.
  gv_text = '완료'.
ELSEIF gv_status = 'N'.
  gv_text = '신규'.
ELSE.
  gv_text = '기타'.
ENDIF.
```

**해설**: 문법 오류가 없어도 업무 의미가 뒤바뀔 수 있습니다. 디버거에서 `gv_status`와 실제로 진입한 분기를 함께 확인합니다.

### D06 — 공통 도메인 재사용 판단

`배송상태`와 `주문상태`가 모두 N/C/X 한 글자를 사용합니다. 다음 중 어느 설계가 더 적절한지 이유를 쓰세요.

1. 하나의 도메인과 하나의 DE를 모두 공유
2. 하나의 도메인을 공유하고 DE는 각각 생성
3. 도메인과 DE를 모두 각각 생성

**정답 예시**: 업무상 허용값과 기술 속성이 정말 같다면 2번이 적절합니다. 공통 도메인으로 CHAR1과 N/C/X를 재사용하고, `주문상태`와 `배송상태`라는 서로 다른 의미·라벨은 별도 DE로 표현합니다.

**해설**: 같은 코드값처럼 보여도 향후 허용값이 달라질 가능성이 크다면 도메인도 분리할 수 있습니다. 핵심은 기술 속성과 업무 의미를 구분해 판단하는 것입니다. 고정값은 화면 입력/F4의 기준이지 임의의 DB 변경을 자동 차단하는 DB 제약조건은 아닙니다.

### D07 — 테이블인가 스트럭처인가

다음 객체를 테이블 또는 스트럭처로 분류하고 이유를 적으세요.

1. 주문을 영구 저장하는 `ZEDU##_ORDER`
2. 주문+고객명+제품명+계산금액을 ALV에 보여주는 `ZEDU##_S_LIST`
3. 팝업에서 잠시 받은 고객ID·제품ID·수량 묶음

**정답**

1. 투명 테이블: DB에 주문을 영구 저장해야 합니다.
2. 스트럭처: JOIN·계산 결과의 모양만 필요하고 별도 저장하지 않습니다.
3. 스트럭처 또는 프로그램 로컬 구조: 실행 중 잠시 사용할 모양만 필요합니다.

**해설**: “데이터를 DB에 계속 보관해야 하는가?”가 첫 판단 기준입니다. 화면 출력 모양과 임시 작업 영역을 별도 테이블로 만들 필요는 없습니다.

## U09. 내부 테이블 선언·APPEND·LOOP

### E09 — 주문 세 건 담기

```abap
REPORT zedu##_u09e.

TYPES: BEGIN OF ty_order,
         order_id TYPE zedu##_ordid,
         prod_id  TYPE zedu_prodid,
         quantity TYPE zedu_qty,
       END OF ty_order.

DATA: gt_order TYPE TABLE OF ty_order,
      gs_order TYPE ty_order.

* TODO 1: 1/P0000001/2, 2/P0000003/10, 3/P0000004/5를 APPEND 하세요.
* TODO 2: LOOP로 주문번호·제품ID·수량을 출력하세요.
```

**정답**

```abap
gs_order-order_id = 1. gs_order-prod_id = 'P0000001'. gs_order-quantity = 2.
APPEND gs_order TO gt_order.
CLEAR gs_order.
gs_order-order_id = 2. gs_order-prod_id = 'P0000003'. gs_order-quantity = 10.
APPEND gs_order TO gt_order.
CLEAR gs_order.
gs_order-order_id = 3. gs_order-prod_id = 'P0000004'. gs_order-quantity = 5.
APPEND gs_order TO gt_order.

LOOP AT gt_order INTO gs_order.
  WRITE: / gs_order-order_id, gs_order-prod_id, gs_order-quantity.
ENDLOOP.
```

**해설**: 한 행을 워크에어리어에 채운 뒤 내부 테이블에 복사합니다. 다음 행을 채우기 전 `CLEAR`하는 습관을 들입니다.

## U10. READ·SORT·MODIFY·DELETE

### E10 — 특정 주문 수량 변경

### D12 — 고객명 포함 주문 건수 TOP

```abap
REPORT zedu##_u12d.

TYPES: BEGIN OF ty_rank,
         cust_id   TYPE zedu_custid,
         cust_name TYPE zedu_custname,
         order_cnt TYPE i,
       END OF ty_rank.

DATA: gt_rank TYPE TABLE OF ty_rank,
      gs_rank TYPE ty_rank.

* TODO: 주문과 고객을 JOIN하여 고객별 주문 건수를 구하고,
*       건수 내림차순으로 정렬해 상위 3명만 조회·출력하세요.
```

**정답**

```abap
SELECT a~cust_id b~cust_name COUNT( * ) AS order_cnt
  FROM zedu##_order AS a
  INNER JOIN zedu_cust AS b ON a~cust_id = b~cust_id
  INTO TABLE gt_rank
  UP TO 3 ROWS
  GROUP BY a~cust_id b~cust_name
  ORDER BY order_cnt DESCENDING.

LOOP AT gt_rank INTO gs_rank.

ENDLOOP.
```

```abap
* TODO 2: 주문번호와 amount를 문자열로 만들어 I 메시지로 표시하세요.
*         (amount는 숫자 타입이라 lv_amt에 옮겨 CONDENSE한 뒤 연결하세요)
  ENDCASE.
ENDFORM.
```

**정답**

```abap
      READ TABLE gt_list INTO gs_list INDEX ps_selfield-tabindex.
      IF sy-subrc <> 0.
        MESSAGE i022(zedu##_msg).
        RETURN.
      ENDIF.

      lv_amt = gs_list-amount.
      CONDENSE lv_amt.
      CONCATENATE '주문' gs_list-order_id '금액' lv_amt
        INTO lv_msg SEPARATED BY space.
      MESSAGE lv_msg TYPE 'I'.
```

**해설**: ALV의 `USER_COMMAND` 콜백은 고전 리스트의 `AT USER-COMMAND` 이벤트와 다른 진입점입니다. 여기서는 콜백 파라미터의 행 인덱스를 사용합니다. `CONCATENATE`는 문자형 인자만 허용하므로 P 타입인 금액은 문자 변수(`lv_amt`)에 옮겨 변환한 뒤 연결합니다.

# Day 3 TP-5 확장 과제 1~6

> 아래 항목은 기본 완성본 이후의 선택 과제입니다. 먼저 조회·등록·수정·삭제 통합 시나리오를 통과한 뒤 시도하세요.
> 기본 과정에서는 취소(X)를 조회·필터용으로만 사용하며, X 주문 수정 차단 규칙을 유지합니다. 확장 과제 1에서만 취소 기능을 명시적으로 추가합니다.

## 확장 1. [주문 취소] 버튼 — N → X

### 목표

- SE41의 `MAIN` 상태에 기능 코드 `FC_CANCEL` 버튼을 추가합니다.
- 신규(N) 주문만 취소(X)로 바꿉니다.
- 완료(C) 또는 이미 취소(X)인 주문은 변경하지 않습니다.

### 힌트 스켈레톤

```abap
" user_command 안
WHEN 'FC_CANCEL'.
  PERFORM cancel_order USING ps_selfield-tabindex.

FORM cancel_order USING pv_index TYPE sytabix.
* TODO 1: 커서 행을 확인하세요.
* TODO 2: 상태 N만 허용하세요.
* TODO 3: DB 상태를 X로 UPDATE하고 COMMIT/ROLLBACK 하세요.
ENDFORM.
```

### 풀이 예시

```abap
FORM cancel_order USING pv_index TYPE sytabix.
  IF pv_index = 0.
    MESSAGE i022(zedu##_msg).
    RETURN.
  ENDIF.

  READ TABLE gt_list INTO gs_list INDEX pv_index.
  IF sy-subrc <> 0.
    MESSAGE i022(zedu##_msg).
    RETURN.
  ENDIF.

  IF gs_list-status <> 'N'.
    MESSAGE '신규 주문만 취소할 수 있습니다.' TYPE 'S' DISPLAY LIKE 'E'.
    RETURN.
  ENDIF.

  UPDATE zedu##_order SET status = 'X'
    WHERE order_id = gs_list-order_id.
  IF sy-subrc = 0.
    COMMIT WORK.
    MESSAGE '주문이 취소되었습니다.' TYPE 'S'.
  ELSE.
    ROLLBACK WORK.
    MESSAGE '주문 취소에 실패했습니다.' TYPE 'S' DISPLAY LIKE 'E'.
  ENDIF.
ENDFORM.
```

### 해설·판정

`user_command`의 공통 후처리인 `PERFORM get_data`와 `ps_selfield-refresh = 'X'`가 목록을 갱신합니다. N 주문은 X로 바뀌고, 그 X 행에서 [수정]을 누르면 기본 완성본의 “취소된 주문은 수정할 수 없습니다” 메시지로 계속 차단되어야 합니다.

## 확장 2. 더블클릭 `&IC1` 주문 상세

### 힌트

더블클릭도 같은 ALV 콜백으로 들어오며 기능 코드는 `&IC1`입니다. `ps_selfield-tabindex`로 행을 읽습니다.

### 풀이 예시

```abap
FORM user_command USING pv_ucomm    LIKE sy-ucomm
                        ps_selfield TYPE slis_selfield.
  DATA: lv_msg   TYPE c LENGTH 200,
        lv_qty_c TYPE c LENGTH 10.

  CASE pv_ucomm.
    WHEN '&IC1'.
      READ TABLE gt_list INTO gs_list INDEX ps_selfield-tabindex.
      IF sy-subrc <> 0.
        MESSAGE i022(zedu##_msg).
        RETURN.
      ENDIF.

      lv_qty_c = gs_list-quantity.     " INT4는 CONCATENATE에 직접 못 쓰므로 변환
      CONDENSE lv_qty_c.
      CONCATENATE '주문:' gs_list-order_id
                  '고객:' gs_list-cust_name
                  '제품:' gs_list-prod_name
                  '수량:' lv_qty_c
                  '상태:' gs_list-status
             INTO lv_msg SEPARATED BY space.
      MESSAGE lv_msg TYPE 'I'.

    " 기존 FC_ADD/FC_EDIT/FC_DEL 분기는 그대로 유지
  ENDCASE.
ENDFORM.
```

### 해설·판정

아무 셀이나 더블클릭했을 때 같은 행의 상세가 I 팝업에 표시되면 성공입니다. 기존 버튼 분기를 삭제하지 말고 `WHEN '&IC1'`만 추가합니다. 수량(INT4) 같은 숫자 필드는 문자 변수로 변환한 뒤 `CONCATENATE`에 넣습니다.

## 확장 3. ALV 금액 합계 — `do_sum`

### 핵심 개념

`do_sum`은 필드 카탈로그의 열 속성입니다. 기본 과정은 `i_structure_name`으로 자동 카탈로그를 만들었으므로, 먼저 표준 FM으로 카탈로그를 받아 AMOUNT 행만 수정합니다.

### 풀이 예시

```abap
FORM show_alv.
  DATA: ls_layout   TYPE slis_layout_alv,
        lt_fieldcat TYPE slis_t_fieldcat_alv,
        ls_fieldcat TYPE slis_fieldcat_alv.

  ls_layout-zebra             = 'X'.
  ls_layout-colwidth_optimize = 'X'.

  CALL FUNCTION 'REUSE_ALV_FIELDCATALOG_MERGE'
    EXPORTING
      i_structure_name = 'ZEDU##_S_LIST'
    CHANGING
      ct_fieldcat      = lt_fieldcat
    EXCEPTIONS
      OTHERS           = 1.
  IF sy-subrc <> 0.
    MESSAGE '필드 카탈로그 생성 오류' TYPE 'S' DISPLAY LIKE 'E'.
    RETURN.
  ENDIF.

  READ TABLE lt_fieldcat INTO ls_fieldcat WITH KEY fieldname = 'AMOUNT'.
  IF sy-subrc = 0.
    ls_fieldcat-do_sum = 'X'.
    MODIFY lt_fieldcat FROM ls_fieldcat INDEX sy-tabix.
  ENDIF.

  CALL FUNCTION 'REUSE_ALV_GRID_DISPLAY'
    EXPORTING
      i_callback_program       = sy-repid
      i_callback_pf_status_set = 'SET_STATUS'
      i_callback_user_command  = 'USER_COMMAND'
      is_layout                = ls_layout
      it_fieldcat              = lt_fieldcat
      i_grid_title             = '주문 목록'
    TABLES
      t_outtab                 = gt_list
    EXCEPTIONS
      program_error            = 1
      OTHERS                   = 2.
ENDFORM.
```

### 해설·판정

`AMOUNT`는 숫자형이므로 합계가 가능합니다. 목록 아래에 금액 합계가 표시되면 성공입니다. `it_fieldcat`을 직접 전달하므로 같은 호출에서 `i_structure_name`은 생략했습니다.

## 확장 4. 상태별 행 색상 — `info_fieldname`

### 준비

SE11의 학생별 출력 스트럭처 `ZEDU##_S_LIST` 끝에 기술용 필드 `LINE_COLOR`(CHAR 4)를 추가하고 활성화합니다. 이 필드는 DB 저장용이 아니라 ALV 제어용입니다.

### 풀이 예시

```abap
" get_data에서 amount 계산과 함께 색상 채우기
LOOP AT gt_list INTO gs_list.
  gs_list-amount = gs_list-quantity * gs_list-price.

  CLEAR gs_list-line_color.
  IF gs_list-status = 'X'.
    gs_list-line_color = 'C610'.   " 빨강 계열
  ELSEIF gs_list-status = 'C'.
    gs_list-line_color = 'C510'.   " 초록 계열
  ENDIF.

  MODIFY gt_list FROM gs_list TRANSPORTING amount line_color.
ENDLOOP.

" show_alv의 레이아웃
ls_layout-info_fieldname = 'LINE_COLOR'.
```

필드 카탈로그를 사용하는 경우 기술 필드를 숨깁니다.

```abap
READ TABLE lt_fieldcat INTO ls_fieldcat WITH KEY fieldname = 'LINE_COLOR'.
IF sy-subrc = 0.
  ls_fieldcat-no_out = 'X'.
  MODIFY lt_fieldcat FROM ls_fieldcat INDEX sy-tabix.
ENDIF.
```

### 해설·판정

`info_fieldname`에는 색상값 자체가 아니라 “각 행의 색상 코드를 담은 필드 이름”을 지정합니다. X 행은 빨강 계열, C 행은 초록 계열로 보이고 N 행은 기본색이면 성공입니다. 색상 코드는 SAP GUI 테마에 따라 조금 다르게 보일 수 있습니다.

## 확장 5. 체크박스 다중 선택 삭제 — `box_fieldname`

### 준비와 주의

`ZEDU##_S_LIST` 끝에 기술용 필드 `SEL`(CHAR 1)을 추가합니다. 기본 삭제 규칙과 동일하게 신규(N)만 삭제합니다. 체크박스 편집값을 내부 테이블에 반영하기 위해 ALV Grid의 `check_changed_data`를 먼저 호출합니다.

### 레이아웃

```abap
ls_layout-box_fieldname = 'SEL'.
```

### 삭제 FORM 예시

```abap
FORM delete_selected.
  DATA: lo_grid    TYPE REF TO cl_gui_alv_grid,
        lv_deleted TYPE i.

  CALL FUNCTION 'GET_GLOBALS_FROM_SLVC_FULLSCR'
    IMPORTING
      e_grid = lo_grid.
  IF lo_grid IS BOUND.
    CALL METHOD lo_grid->check_changed_data.
  ENDIF.

  LOOP AT gt_list INTO gs_list WHERE sel = 'X'.
    IF gs_list-status <> 'N'.
      CONTINUE.
    ENDIF.

    DELETE FROM zedu##_order WHERE order_id = gs_list-order_id.
    IF sy-subrc = 0.
      lv_deleted = lv_deleted + 1.
    ENDIF.
  ENDLOOP.

  IF lv_deleted > 0.
    COMMIT WORK.
    MESSAGE '선택한 신규 주문을 삭제했습니다.' TYPE 'S'.
  ELSE.
    ROLLBACK WORK.
    MESSAGE '삭제할 신규 주문을 선택하세요.' TYPE 'S' DISPLAY LIKE 'E'.
  ENDIF.
ENDFORM.
```

`USER_COMMAND`에는 별도 기능 코드(예: `FC_MDEL`)를 연결합니다.

```abap
WHEN 'FC_MDEL'.
  PERFORM delete_selected.
```

### 해설·판정

N 행 두 개와 C/X 행을 함께 체크했을 때 N 행만 삭제되어야 합니다. 교육용 단순화를 따른 예제이며, 실제 대량 변경은 잠금·권한·LUW·부분 실패 정책을 먼저 설계해야 합니다.

## 확장 6. `get_data`를 new Open SQL로 리팩터링

### 풀이 예시

```abap
FORM get_data_new.
  SELECT FROM zedu##_order AS a
    INNER JOIN zedu_cust AS b ON a~cust_id = b~cust_id
    INNER JOIN zedu_prod AS c ON a~prod_id = c~prod_id
    FIELDS a~order_id, a~cust_id, b~cust_name,
           a~prod_id, c~prod_name, c~price,
           a~quantity, a~status, a~order_date, a~remark, a~ernam
    WHERE a~order_id   IN @s_ordid
      AND a~cust_id    IN @s_cust
      AND a~order_date IN @s_date
    INTO CORRESPONDING FIELDS OF TABLE @gt_list.

  IF p_stat IS NOT INITIAL.
    DELETE gt_list WHERE status <> p_stat.
  ENDIF.

  LOOP AT gt_list ASSIGNING FIELD-SYMBOL(<ls_list>).
    <ls_list>-amount = <ls_list>-quantity * <ls_list>-price.
  ENDLOOP.

  SORT gt_list BY order_id.
ENDFORM.
```

### 해설·판정

신문법의 표식은 콤마 필드 목록, ABAP 호스트 변수 앞 `@`, `FIELDS ... INTO` 순서, 필드 심볼 인라인 선언입니다. 기존 `get_data`와 같은 선택조건에서 건수·필드값·금액이 모두 같아야 성공입니다. 기본 본문 코드는 classic 문법으로 유지하고 이 FORM은 비교·자습용으로 둡니다.
